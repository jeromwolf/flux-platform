## 1장. 과업 개요

### 1.1 상위 과제 및 과업 배경

#### 1.1.1 상위 과제

본 과업은 **"선박해양 디지털 전환 지원을 위한 디지털서비스 플랫폼 개발"** 과제의 세부 과업입니다. 상위 과제는 해양수산부가 추진하는 5개년(2026~2030) 규모의 국가 R&D 사업으로, 선박해양 분야의 디지털 전환을 지원하기 위한 통합 서비스 플랫폼 구축을 목표로 합니다.

| 항목 | 내용 |
|------|------|
| 상위 과제명 | 선박해양 디지털 전환 지원을 위한 디지털서비스 플랫폼 개발 |
| 총괄 기관 | 한국해양과학기술원 부설 선박해양플랜트연구소 (KRISO) |
| 사업 기간 | 5개년 (2026~2030) |
| 본 세부 과업 | 대화형 해사서비스 플랫폼 KG 모델 설계 연구 |
| 본 과업 단계 | 1차년도 (설계 + 개념 증명) |

#### 1.1.2 과업 배경

해사(Maritime) 분야는 선박 운항, 항만 물류, 해양 기상, 해양 안전, 수산, 연구 실험 등 다양한 영역에서 대량의 이질적(heterogeneous) 데이터가 생성되고 있습니다. 그러나 이들 데이터는 각 기관별로 독립적인 시스템에 분산 관리되고 있어, 데이터 간의 유기적 연계와 통합 활용이 어려운 실정입니다.

특히 KRISO(한국해양과학기술원 부설 선박해양플랜트연구소)는 해양공학수조, 선형시험수조, 빙해수조, 심해공학수조, 파력발전 실해역 시험장, 고압챔버, 캐비테이션터널, 선박운항시뮬레이터 등 **8종의 세계적 수준 시험시설**을 보유하고 있습니다. 이들 시설에서 생산되는 실험 데이터(센서 계측값, 영상, 파형 데이터 등)와 외부 해사 데이터(AIS, 위성영상, 해양기상, 사고 이력 등) 간의 관계를 체계적으로 관리할 필요성이 대두되었습니다.

이러한 배경에서 본 과업은 **지식그래프(Knowledge Graph)** 기술을 활용하여 해사 도메인의 멀티모달 메타데이터를 관계 중심으로 구조화하고, **자연어 대화**를 통해 비전문가도 해사 정보에 접근할 수 있는 플랫폼의 기반을 설계하는 것을 목적으로 합니다.

> **1차년도 범위 제한:** 본 과업은 "설계 + PoC" 단계로서, **멀티모달 데이터 자체의 RAG(검색 증강 생성) 처리가 아닌, 멀티모달 메타데이터 간의 '관계'를 지식그래프로 구조화하는 것**이 핵심 범위입니다.

## 2장. 해사 도메인 온톨로지 설계 결과

### 2.1 온톨로지 설계 방법론

본 과업에서 설계한 해사 도메인 온톨로지는 세 가지 핵심 개념을 중심으로 해사 분야의 지식을 형식적(formal)으로 정의합니다.

| 핵심 개념 | 한국어명 | 설명 | 예시 |
|-----------|----------|------|------|
| ObjectType | 객체 유형 | 도메인 내 식별 가능한 개체 유형 | 선박(Vessel), 항만(Port), 시험시설(TestFacility) |
| LinkType | 관계 유형 | 객체 간의 의미적 연결 유형 | 정박(DOCKED_AT), 항해중(ON_VOYAGE), 수행시설(CONDUCTED_AT) |
| PropertyDefinition | 속성 정의 | 객체 또는 관계에 부여되는 특성 | 선박명(name), 위치(location), 실험일자(date) |

**설계 원칙:**

| 원칙 | 설명 |
|------|------|
| **메타데이터 우선** | 바이너리 데이터(영상, 센서 원시값 등)는 외부 저장소에 두고, 그래프에는 메타데이터와 관계만 저장 |
| **국제 표준 준수** | IHO S-100 범용수로데이터모델(Universal Hydrographic Data Model), ISO 19100 시리즈 준수 |
| **점진적 확장** | PoC 핵심 범위에서 시작하여 2차년도 이후 점진적으로 확장 가능한 구조 |
| **다국어 지원** | 한국어/영어 이중 속성 패턴 적용 (예: name / nameEn) |
| **접근 제어 내장** | 데이터 분류 등급에 따른 접근 제어를 그래프 구조 내에 내장 |

#### 2.1.1 속성 정의 체계

객체 유형과 관계 유형에 부여되는 속성은 다음과 같은 타입 체계를 지원합니다.

| 속성 타입(PropertyType) | 한국어명 | 설명 | 사용 예시 |
|------------------------|----------|------|-----------|
| STRING | 문자열 | 텍스트 데이터 | 선박명(name), 시설명(facilityName) |
| INTEGER | 정수 | 정수형 숫자 | MMSI 번호, IMO 번호 |
| FLOAT | 실수 | 실수형 숫자 | 위도(latitude), 경도(longitude), 파고(waveHeight) |
| BOOLEAN | 참/거짓 | 논리값 | 활성 여부(isActive) |
| DATE | 날짜 | 날짜 정보 | 건조일(buildDate) |
| DATETIME | 날짜시간 | 날짜+시간 정보 | 관측시각(timestamp), 입항시각(arrivalTime) |
| POINT | 좌표점 | 지리 좌표 (위도/경도) | 선박위치(location), 사고위치(incidentLocation) |
| LIST<STRING> | 문자열 목록 | 문자열 배열 | 화물 목록(cargoList), 태그(tags) |
| LIST<FLOAT> | 실수 목록 | 실수 배열 | 임베딩 벡터(embedding) |

각 속성 정의에는 필수 여부(required), 기본키 여부(primaryKey), 인덱싱 여부(indexed), 유일성 여부(unique), 값 범위(minValue/maxValue), 패턴(pattern) 등의 제약조건을 선언적으로 부여할 수 있습니다.

### 2.2 11개 엔티티 그룹 분류

126개 객체 유형(ObjectType)을 의미적 유사성에 따라 11개 상위 그룹으로 분류하였습니다. 각 그룹은 해사 도메인의 주요 지식 영역을 대표합니다.

| # | 그룹명 | 한국어명 | 객체 유형 수 | 주요 객체 유형 예시 |
|---|--------|----------|-------------|-------------------|
| 1 | PhysicalEntity | 물리적 객체 | 25개 | 선박(Vessel), 항만(Port), 부두(Berth), 항로(Channel), 화물(Cargo), 센서(Sensor) |
| 2 | SpatialEntity | 공간 정보 | 5개 | 해역(SeaArea), 배타적경제수역(EEZ), 영해(TerritorialSea), 지리좌표(GeoPoint) |
| 3 | TemporalEntity | 시간/사건 정보 | 16개 | 항해(Voyage), 입항(PortCall), 항적구간(TrackSegment), 해양사고(Incident), 활동(Activity) |
| 4 | InformationEntity | 정보 객체 | 16개 | 규정(Regulation), 문서(Document), 사고보고서(AccidentReport), 데이터원본(DataSource), 서비스(Service) |
| 5 | Observation | 관측/계측 | 6개 | SAR관측(SARObservation), AIS관측(AISObservation), 기상관측(WeatherObservation) |
| 6 | Agent | 행위자 | 8개 | 기관(Organization), 정부기관(GovernmentAgency), 연구기관(ResearchInstitute), 선원(CrewMember) |
| 7 | PlatformResource | 플랫폼 자원 | 10개 | 워크플로우(Workflow), AI모델(AIModel), 데이터파이프라인(DataPipeline), AI에이전트(AIAgent) |
| 8 | MultimodalData | 멀티모달 데이터 | 6개 | AIS데이터(AISData), 위성영상(SatelliteImage), 레이더영상(RadarImage), 센서계측(SensorReading) |
| 9 | MultimodalRepresentation | 멀티모달 표현 | 4개 | 시각임베딩(VisualEmbedding), 텍스트임베딩(TextEmbedding), 궤적임베딩(TrajectoryEmbedding), 융합임베딩(FusedEmbedding) |
| 10 | KRISO | KRISO 연구 | 16개 | 실험(Experiment), 시험시설(TestFacility), 축척모형선(ModelShip), 실험데이터셋(ExperimentalDataset), 계측(Measurement) |
| 11 | RBAC | 접근 제어 | 4개 | 사용자(User), 역할(Role), 데이터분류(DataClass), 권한(Permission) |

> **참고:** 일부 세부 객체 유형은 상위 유형의 특수화(specialization)입니다. 예를 들어 화물선(CargoShip), 유조선(Tanker), 어선(FishingVessel), 여객선(PassengerShip)은 모두 선박(Vessel)의 하위 유형입니다.

### 2.3 KRISO 시험시설 모델링 결과

KRISO가 보유한 8종의 세계적 수준 시험시설을 지식그래프 내에서 체계적으로 모델링하였습니다. 각 시설은 고유한 객체 유형으로 정의되며, 실험, 축척모형선, 계측 데이터와의 관계를 통해 연구 데이터의 맥락(context)을 보존합니다.

#### 2.3.1 8종 시험시설 모델링

| # | 시설명 | 객체 유형 | 시설 식별자 | 주요 시험 분야 |
|---|--------|-----------|------------|---------------|
| 1 | 해양공학수조 | OceanEngineeringBasin | KRISO-OEB | 파랑, 바람, 조류 복합 환경 시험, 내항성능 평가 |
| 2 | 선형시험수조 / 대형예인수조 | TowingTank | KRISO-TT | 선박 저항 및 추진 성능 시험 |
| 3 | 빙해수조 | IceTank | KRISO-IT | 빙해역 선박/구조물 성능 시험, 쇄빙 능력 평가 |
| 4 | 심해공학수조 | DeepOceanBasin | KRISO-DOEB | 심해 구조물 거동 시험, 라이저/계류 시스템 |
| 5 | 파력발전 실해역 시험장 | WaveEnergyTestSite | KRISO-WETS | 파력 에너지 변환 장치 실해역 성능 검증 |
| 6 | 고압챔버 | HyperbaricChamber | KRISO-HC | 수중 장비/소재 고압 환경 시험 |
| 7 | 캐비테이션터널 (대형/중형/고속) | CavitationTunnel | KRISO-CT | 프로펠러/선체 캐비테이션 현상 시험 |
| 8 | 선박운항시뮬레이터 | BridgeSimulator | KRISO-FMBS | 선박 조종 시뮬레이션, 운항 훈련, 항만 설계 검증 |

**시설별 상세 설명:**

- **해양공학수조(OceanEngineeringBasin):** 파랑, 바람, 조류를 동시에 발생시킬 수 있는 복합 환경 시험 시설입니다. 주로 선박 및 해양구조물의 내항성능, 계류 시스템 거동, 파랑중 운동 응답 등을 평가합니다. 온톨로지에서는 내항 성능(Seakeeping), 조종 성능(Maneuvering) 계측 유형과 연결됩니다.

- **선형시험수조/대형예인수조(TowingTank):** 선박 모형을 예인하여 저항과 추진 성능을 측정하는 시설입니다. 저항 시험(Resistance), 추진 성능(Propulsion) 계측 유형이 이 시설에서 주로 생산됩니다. 세계 최대 규모의 시설 중 하나로, 실선에 가까운 조건의 시험이 가능합니다.

- **빙해수조(IceTank):** 인공 빙판을 생성하여 극지 운항 선박의 쇄빙 능력, 빙해 저항 등을 평가합니다. 빙해 성능(IcePerformance) 계측 유형과 연결되며, 북극항로 개척에 따라 중요성이 높아지는 시설입니다.

- **심해공학수조(DeepOceanBasin):** 심해 환경을 모사하여 해양구조물(부유식 생산설비, 해저 파이프라인 등)의 거동을 평가합니다. 구조 응답(StructuralResponse) 계측 유형과 연결됩니다.

- **파력발전 실해역 시험장(WaveEnergyTestSite):** 실제 바다에서 파력 에너지 변환 장치의 성능을 검증하는 개방형 시험장입니다. 다른 시설과 달리 실해역 환경에서 운영됩니다.

- **고압챔버(HyperbaricChamber):** 수심 수백 미터의 수압 환경을 재현하여 수중 장비와 소재의 내압 성능을 시험합니다.

- **캐비테이션터널(CavitationTunnel):** 프로펠러와 선체 주변의 캐비테이션 현상을 관찰하고 측정하는 시설입니다. 대형, 중형, 고속 3종의 터널이 있으며, 온톨로지에서는 LargeCavitationTunnel, MediumCavitationTunnel, HighSpeedCavitationTunnel로 세분화하여 모델링하였습니다.

- **선박운항시뮬레이터(BridgeSimulator):** 실제 선교(Bridge)를 재현한 풀미션(Full Mission) 시뮬레이터로, 선박 조종 훈련, 항만 설계 검증, 교통 흐름 분석 등에 활용됩니다.

#### 2.3.2 시험시설 중심 관계 모델

KRISO 시험시설을 중심으로 다음과 같은 관계 구조가 형성됩니다.

```
(실험:Experiment)─[수행시설:CONDUCTED_AT]──>(시험시설:TestFacility)
        │
        ├─[시험대상:TESTED]──────────────>(축척모형선:ModelShip)
        │                                       │
        │                                [원형선박:MODEL_OF]──>(선박:Vessel)
        │
        ├─[결과데이터:PRODUCED]───────────>(실험데이터셋:ExperimentalDataset)
        │                                       │
        │                                [원시계측:HAS_RAW_DATA]──>(계측:Measurement)
        │
        ├─[시험조건:UNDER_CONDITION]──────>(시험조건:TestCondition)
        │
        ├─[촬영영상:RECORDED_VIDEO]──────>(비디오클립:VideoClip)
        │
        ├─[센서계측:MEASURED_DATA]────────>(센서계측값:SensorReading)
        │
        └─[관련규정:COMPLIES_WITH]────────>(규정:Regulation)
```

이 관계 구조를 통해 연구자는 다음과 같은 질문에 답할 수 있습니다:
- "해양공학수조에서 2025년에 수행된 실험 목록은?"
- "특정 축척모형선이 어떤 시설에서 어떤 조건으로 시험되었는지?"
- "동일 선박의 다른 시설(예인수조 vs 캐비테이션터널) 시험 결과 비교"
- "특정 시험조건(파고, 풍속 등)에서의 실험 데이터 검색"

#### 2.3.3 계측 데이터 유형

KRISO 시험시설에서 생산되는 계측 데이터를 6개 유형으로 분류하여 모델링하였습니다.

| 계측 유형 | 객체 유형 | 관련 시설 | 주요 계측 항목 |
|-----------|-----------|-----------|---------------|
| 저항 시험 | Resistance | 선형시험수조 | 저항력, 속도, 트림 |
| 추진 성능 | Propulsion | 선형시험수조 | 추력, 토크, 효율 |
| 조종 성능 | Maneuvering | 해양공학수조, 시뮬레이터 | 선회 반경, 타각, 횡경사 |
| 내항 성능 | Seakeeping | 해양공학수조 | 종동요, 횡동요, 상하가속도 |
| 빙해 성능 | IcePerformance | 빙해수조 | 쇄빙 저항, 빙 두께, 빙 강도 |
| 구조 응답 | StructuralResponse | 심해공학수조 | 변형률, 응력, 피로 하중 |

### 2.4 핵심 관계 패턴

83개 관계 유형(LinkType)을 의미적 범주에 따라 7가지 패턴으로 분류하였습니다.

#### 패턴 1: 물리적 위치/연결 관계 (6개)

선박, 항만, 해역 간의 물리적 위치와 연결을 나타냅니다.

| 관계 유형 | 시작 객체 | 종료 객체 | 의미 |
|-----------|-----------|-----------|------|
| LOCATED_AT (위치) | 선박 | 해역 | 선박의 현재 위치 |
| DOCKED_AT (정박) | 선박 | 부두 | 선박의 접안 상태 |
| ANCHORED_AT (정박대기) | 선박 | 정박지 | 선박의 정박대기 상태 |
| HAS_FACILITY (시설보유) | 항만 | 항만시설 | 항만의 시설 구성 |
| CONNECTED_VIA (연결항로) | 항만 | 수로 | 항만 접근 항로 |
| CONNECTS (연결) | 수로 | 해역 | 수로의 해역 연결 |

#### 패턴 2: 운항/물류 관계 (6개)

선박의 항해, 화물 운송, 운항 활동을 나타냅니다.

| 관계 유형 | 의미 |
|-----------|------|
| ON_VOYAGE (항해중) | 선박이 항해를 수행 |
| FROM_PORT / TO_PORT (출발항/도착항) | 항해의 출발지와 목적지 |
| CONSISTS_OF (구성) | 항해를 구성하는 항적 구간 |
| CARRIES (운송) | 선박이 운송하는 화물 |
| PERFORMS (수행) | 선박이 수행하는 활동(적하, 양하, 급유 등) |

#### 패턴 3: 관측/감시 관계 (8개)

센서, 관측 기록, 탐지/식별 결과 간의 관계를 나타냅니다.

| 관계 유형 | 의미 |
|-----------|------|
| PRODUCES (생산) | 센서가 관측 기록을 생산 |
| DEPICTS / DETECTED / IDENTIFIED | 관측 기록이 선박을 포착/탐지/식별 |
| OBSERVED_AT (관측위치) | 관측 기록의 지리적 위치 |
| HAS_EMBEDDING (임베딩연결) | 관측 기록의 벡터 표현 연결 |
| MATCHED_WITH / SAME_ENTITY | 교차 모달 관측 간 매칭 |

#### 패턴 4: 환경/사고 관계 (5개)

해양 기상, 사고, 규정 위반 간의 인과 관계를 나타냅니다.

| 관계 유형 | 의미 |
|-----------|------|
| AFFECTS (영향) | 기상 조건이 해역에 영향 |
| CAUSED_BY (원인) | 사고의 기상 원인 |
| OCCURRED_AT (발생위치) | 사고 발생 위치 |
| INVOLVES (관련선박) | 사고에 관련된 선박 |
| VIOLATED (규정위반) | 사고와 관련된 규정 위반 |

#### 패턴 5: 규정/문서 관계 (5개)

해사 규정, 문서, 기관 간의 관계를 나타냅니다.

| 관계 유형 | 의미 |
|-----------|------|
| APPLIES_TO (적용대상) | 규정이 적용되는 선박 유형 |
| ENFORCED_BY (집행기관) | 규정을 집행하는 기관 |
| DESCRIBES (기술) | 문서가 사고를 기술 |
| ISSUED_BY (발행기관) | 문서를 발행한 기관 |
| COMPLIES_WITH (준수) | 규정 준수 관계 |

#### 패턴 6: KRISO 연구 관계 (12개)

KRISO 시험시설, 실험, 데이터셋 간의 연구 관계를 나타냅니다. (2.3.2절 참조)

| 관계 유형 | 의미 |
|-----------|------|
| CONDUCTED_AT (수행시설) | 실험 수행 시설 |
| TESTED (시험대상) | 실험 대상 축척모형선 |
| PRODUCED (결과생산) | 실험 결과 데이터셋 생산 |
| UNDER_CONDITION (시험조건) | 실험 조건 |
| MODEL_OF (원형) | 축척모형선의 원형 선박 |
| HAS_RAW_DATA (원시데이터) | 데이터셋의 원시 계측 기록 |
| RECORDED_VIDEO / MEASURED_DATA | 실험 영상/센서 데이터 |
| BASED_ON_CHART / USES_SATELLITE / HAS_AIS_DATA | 실험의 참조 데이터 |

#### 패턴 7: 플랫폼/서비스/멀티모달 관계 (41개)

서비스 간 데이터 흐름, 워크플로우 구성, AI 에이전트 동작, 멀티모달 데이터 연결 등 플랫폼 내부 및 데이터 리니지 관계를 나타냅니다. (상세는 부록 A 참조)

### 2.5 IHO S-100 표준 매핑

IHO S-100 범용수로데이터모델(Universal Hydrographic Data Model)과의 호환성을 확보하기 위해 다음과 같은 매핑 전략을 수립하였습니다.

| S-100 개념 | 본 온톨로지 매핑 | 설명 |
|------------|-----------------|------|
| 지물 유형(Feature Type) | 객체 유형(ObjectType) | 해역, 항만, 선박 등 도메인 개체 |
| 지물 연관(Feature Association) | 관계 유형(LinkType) | 개체 간 의미적 연결 |
| 정보 유형(Information Type) | 속성 정의(PropertyDefinition) | 개체에 부여되는 특성 |
| 코드 목록(Code List) | 열거형(Enum) 속성 | 선박 유형, 화물 유형 등 분류값 |

**관련 IHO 제품 규격(Product Specification) 분석:**

| 규격 번호 | 규격명 | KRISO 연관도 |
|-----------|--------|-------------|
| S-101 | 전자해도(ENC) | 높음 - 해도 데이터 연계 |
| S-104 | 수위 정보 | 중간 - 시험시설 환경 조건 |
| S-111 | 해류 정보 | 높음 - 내항/조종 성능 시험 조건 |
| S-124 | 항행경보 | 중간 - 안전 관련 정보 연계 |
| S-421 | 항로 정보 교환 | 낮음 - 2차년도 이후 연계 |

**OWL 온톨로지 구현 (n10s 통합):**

S-100 매핑 전략의 실제 구현으로, 전체 해사 온톨로지를 OWL 2 Turtle 형식으로 공식화하고 Neosemantics (n10s) 플러그인을 통해 Neo4j에 임포트하는 파이프라인을 구축하였습니다.

| 구현 항목 | 내용 |
|-----------|------|

| 단위 테스트 | `tests/test_n10s.py` — 35개 테스트 전수 통과 |

### 2.6 PoC 핵심 범위와 확장 범위

126개 객체 유형은 전체 설계 범위이며, PoC에서는 이 중 핵심적인 약 40~50개 객체 유형을 구현하여 검증하였습니다. 나머지 객체 유형은 설계만 완료된 상태로, 2차년도 이후 실제 데이터 확보에 따라 순차적으로 구현됩니다.

## 4장. PoC 시연 결과

본 장에서는 PoC 프로토타입의 주요 기능별 시연 결과를 기술합니다. PoC는 1차년도 설계 결과의 기술적 실현 가능성을 검증하기 위한 목적으로 구현되었습니다.

PoC 시연은 다음 4개 핵심 기능을 대상으로 수행하였습니다.

| # | 시연 기능 | 관련 요구사항 | 목적 |
|---|-----------|-------------|------|
| 1 | 자연어 질의 | SFR-001 | 한국어 자연어로 지식그래프를 탐색할 수 있는가? |
| 2 | 지식그래프 시각화 | SFR-002, SFR-003 | 데이터 간 관계를 시각적으로 탐색할 수 있는가? |
| 3 | 접근 제어 | SEC-001 | 역할에 따른 데이터 접근 제한이 동작하는가? |
| 4 | 데이터 수집 | SFR-004 | 외부 해사 데이터를 자동 수집하여 그래프에 적재할 수 있는가? |

### 4.1 자연어 질의 시연

#### 4.1.1 시연 구성

한국어 자연어 질의를 지식그래프 질의 언어(Cypher)로 자동 변환하여 결과를 반환하는 기능을 시연하였습니다.

```
사용자 질의 (한국어)
    |
    v
[한국어 해사용어 사전] -- 객체/관계 인식 (105개 동의어, 22개 관계 키워드)
    |
    v
[대규모 언어 모델] -- Cypher 쿼리 생성 (Ollama qwen2.5:7b)
    |
    v
[Neo4j 실행] -- 쿼리 실행 및 결과 반환
    |
    v
[대규모 언어 모델] -- 자연어 응답 합성
    |
    v
사용자 응답 (한국어)
```

#### 4.1.2 시연 질의 및 결과 예시

| # | 사용자 질의 (한국어) | 인식 결과 | 생성 Cypher (요약) | 비고 |
|---|---------------------|-----------|-------------------|------|
| 1 | "부산항 근처 선박 알려줘" | Port(부산항), Vessel | MATCH (v:Vessel)-[]->(p:Port) WHERE p.name CONTAINS '부산' | 항만-선박 관계 탐색 |
| 2 | "KRISO 해양공학수조에서 한 실험은?" | TestFacility(해양공학수조), Experiment | MATCH (e:Experiment)-[:CONDUCTED_AT]->(f:TestFacility) WHERE f.name CONTAINS '해양공학' | 시험시설-실험 관계 탐색 |
| 3 | "컨테이너선 목록" | Vessel(vesselType=ContainerShip) | MATCH (v:Vessel) WHERE v.vesselType = 'ContainerShip' | 속성 필터 질의 |
| 4 | "최근 해양사고 현황" | Incident | MATCH (i:Incident) RETURN i ORDER BY i.date DESC | 시간 정렬 질의 |
| 5 | "부산항에 정박 중인 선박" | Port(부산), DOCKED_AT, Vessel | MATCH (v:Vessel)-[:DOCKED_AT]->()-[:HAS_FACILITY]->(:Port {name: '부산'}) | 다중 관계 탐색 |

#### 4.1.3 한국어 특화 처리

한국어 자연어 질의의 정확도를 높이기 위해 다음과 같은 전처리를 수행합니다.

| 처리 영역 | 항목 수 | 예시 |
|-----------|---------|------|
| 객체 유형 동의어 | 105개 | "선박", "배", "함선" -> Vessel |
| 관계 유형 키워드 | 22개 | "정박", "접안" -> DOCKED_AT |
| 속성값 매핑 | 다수 | "컨테이너선" -> vesselType: ContainerShip |
| 한국어 조사 처리 | - | "부산항에", "선박을", "실험은" 등 조사 분리 |

### 4.2 지식그래프 시각화 시연

#### 4.2.1 시연 구성

Cytoscape.js 기반의 인터랙티브 그래프 시각화 도구를 통해 지식그래프의 구조를 직관적으로 탐색할 수 있습니다. 시각화 도구는 경량 웹 기반으로 구현되어 별도의 설치 없이 웹 브라우저에서 바로 사용할 수 있으며, Python 표준 라이브러리 기반의 REST API 서버와 연동됩니다.

| 기능 | 설명 |
|------|------|
| 부분그래프 탐색 | 특정 객체 유형(예: 선박, 항만, 시험시설)의 노드와 관계를 시각화 |
| 이웃 확장 | 특정 노드 클릭 시 연결된 이웃 노드를 확장 표시 |
| 한국어/영어 검색 | 노드 이름을 한국어 또는 영어로 검색 |
| 스키마 정보 조회 | 사용 가능한 객체 유형 및 관계 유형 목록 조회 |
| 속성 상세 보기 | 노드 클릭 시 속성 정보 팝업 표시 |

#### 4.2.2 REST API 엔드포인트

| 엔드포인트 | 기능 |
|-----------|------|
| `/api/subgraph?label=Vessel&limit=50` | 객체 유형별 부분그래프 조회 |
| `/api/neighbors?nodeId=xxx` | 특정 노드의 이웃 확장 |
| `/api/search?q=부산` | 이름 기반 노드 검색 |
| `/api/schema` | 사용 가능한 객체 유형 및 관계 유형 조회 |

### 4.3 접근 제어 시연

#### 4.3.1 역할 체계

5개 역할을 정의하여 데이터 접근 범위를 차등화하였습니다.

| 역할 | 설명 | 데이터 접근 범위 |
|------|------|-----------------|
| 관리자(admin) | 시스템 관리자 | 전체 데이터 접근 |
| 연구원(researcher) | KRISO 연구원 | 연구 데이터, 실험 데이터 접근 |
| 운항관리자(operator) | 운항 관리 담당 | 운항 데이터, 항만 데이터 접근 |
| 분석가(analyst) | 데이터 분석 담당 | 공개 데이터, 통계 데이터 접근 |
| 일반사용자(public) | 일반 이용자 | 공개 데이터만 접근 |

#### 4.3.2 데이터 분류 등급

연구기관 환경에 맞는 4단계 데이터 분류 등급을 적용하였습니다.

| 등급 | 한국어명 | 설명 | 접근 가능 역할 |
|------|----------|------|---------------|
| INTERNAL | 내부전용 | KRISO 내부 연구 데이터 | 관리자, 연구원 |
| RESTRICTED | 제한공개 | 협력기관 공유 데이터 | 관리자, 연구원, 운항관리자 |
| CONDITIONAL | 조건부공개 | 승인 후 공개 가능 데이터 | 관리자, 연구원, 운항관리자, 분석가 |
| PUBLIC | 공개 | 대민 서비스 공개 데이터 | 전체 역할 |

#### 4.3.3 접근 제어 동작 원리

접근 제어는 쿼리 수준(query-level)에서 동작합니다. 사용자가 질의를 실행하면, 해당 사용자의 역할(Role)에 따라 접근 가능한 데이터 분류 등급을 확인하고, Cypher 쿼리에 필터 조건을 자동으로 주입(injection)합니다.

```
(사용자:User)─[역할부여:HAS_ROLE]──>(역할:Role)─[접근허용:CAN_ACCESS]──>(데이터분류:DataClass)
(노드)────────[분류지정:CLASSIFIED_AS]──>(데이터분류:DataClass)
```

예를 들어, "일반사용자" 역할로 지식그래프를 질의하면 "공개(PUBLIC)" 등급의 데이터만 반환되며, "연구원" 역할로 질의하면 "내부전용(INTERNAL)"까지의 데이터가 반환됩니다.

### 4.4 데이터 수집 시연

#### 4.4.1 4종 데이터 수집기

| 수집기 | 대상 기관/시스템 | 수집 항목 | 비고 |
|--------|----------------|-----------|------|
| KRISO 논문 수집기 | KRISO ScholarWorks | 논문 제목, 저자, 초록, DOI, 발행일 | 학술 데이터베이스 크롤링 |
| KRISO 시설 수집기 | KRISO 웹사이트 | 시설 제원, 운영 정보 | 시험시설 정보 수집 |

#### 4.4.2 관계 추출

수집된 텍스트 데이터에서 객체 유형 간 관계를 자동으로 추출하는 관계 추출기를 함께 구현하였습니다. 이를 통해 비정형 텍스트 데이터로부터 지식그래프의 관계 정보를 반자동으로 생성할 수 있습니다.

#### 4.4.3 수집기 공통 기능

모든 데이터 수집기는 공통 기반 클래스를 상속하며, 다음과 같은 공통 기능을 제공합니다.

| 공통 기능 | 설명 |
|-----------|------|
| 요청 간 지연(delay) | 대상 서버 부하 방지를 위한 요청 간격 설정 |
| 재시도(retry) 로직 | 네트워크 오류 시 자동 재시도 (지수 백오프) |
| 오류 로깅 | 수집 과정의 오류를 체계적으로 기록 |
| 결과 캐싱 | 중복 수집 방지를 위한 결과 캐싱 |
| JSON/CSV 내보내기 | 수집 결과를 다양한 형식으로 내보내기 |

> **참고:** 모든 데이터 수집기는 대상 서버의 부하를 고려하여 요청 간 지연(delay)을 포함합니다. 수집기 실행 주기 및 데이터 갱신 전략은 2차년도 ETL 파이프라인 구현 시 확정될 예정입니다.

3차년도는 2차년도에 구축된 프로토타입을 실무 환경에서 활용 가능한 MVP(최소 기능 제품)로 통합하는 단계입니다. 외부 기관 데이터를 통합하고, 사용자 피드백을 반영하여 시스템의 실용성을 높이는 것이 핵심입니다.

| 산출물 8종 | 8종 초안 작성 완료 (실 과업 수행 후 최종본 갱신 예정) |
| 온톨로지 설계 | 126개 객체 유형, 83개 관계 유형, 11개 그룹 분류체계 |
| KRISO 시험시설 모델링 | 8종 시험시설 전수 모델링 완료 |
| PoC 시연 | 자연어 질의, 시각화, 접근 제어, 데이터 수집 4개 기능 시연 |
| 테스트 | 1,095개 단위 테스트 전수 통과 (100%) |

### 8.2 핵심 성과

1. **KRISO 시험시설 8종을 포괄하는 해사 도메인 온톨로지**를 설계하여, KRISO 연구 데이터를 시험시설, 실험, 축척모형선, 계측 데이터 간의 관계 맥락 속에서 관리할 수 있는 지식 구조를 확보하였습니다.

2. **한국어 자연어 질의로 지식그래프를 탐색**할 수 있는 가능성을 PoC를 통해 확인하여, 비전문가도 해사 정보에 접근할 수 있는 대화형 서비스의 기술적 기반을 마련하였습니다.

3. **IHO S-100 국제 표준과의 매핑 전략**을 수립하여, 향후 국제 해사 데이터 교환 및 표준 준수의 기반을 확보하였습니다.

4. **OWL 온톨로지 + Neosemantics (n10s) 통합 구현**으로 해사 온톨로지를 국제 표준 OWL 형식(1,433 트리플)으로 공식화하고, n10s 플러그인을 통한 Neo4j 임포트 파이프라인을 구축하여 "분석 → 매핑 → 구현"의 전 과정을 1차년도에 완료하였습니다.

5. **2차년도 구축에 필요한 상세 설계**(ETL 파이프라인, 자연어 탐색 아키텍처, 분류체계)를 완료하여, 2차년도 착수 시 즉시 구현에 집중할 수 있는 기반을 마련하였습니다.

### 8.3 향후 방향

1차년도에 확립된 온톨로지 설계와 지식그래프 스키마를 기반으로, 2차년도에는 다음 세 가지 핵심 목표를 추진할 계획입니다.

**첫째**, 실제 KRISO 시험시설 운영 데이터와 실험 데이터를 지식그래프에 적재하여, 1차년도 설계의 실용성을 검증하고 온톨로지를 실데이터 기반으로 정제합니다.

**둘째**, GraphRAG(그래프 기반 검색 증강 생성) 파이프라인을 구현하여, 연구자가 자연어 대화만으로 복잡한 연구 데이터를 탐색하고 인사이트를 발견할 수 있는 환경을 구축합니다.

**셋째**, 프로덕션 배포를 통해 KRISO 연구원이 실무 환경에서 플랫폼을 활용할 수 있도록 하고, 사용자 피드백을 수집하여 3차년도 MVP 통합에 반영합니다.

특히 KRISO 8종 시험시설에서 생산되는 실험 데이터와 외부 해사 데이터(AIS, 위성영상, 해양기상 등)의 유기적 연계를 통해, **크로스모달 탐색**이라는 차별화된 가치를 실현하는 것이 5개년 과제의 중장기 목표입니다. 이를 통해 분산된 해사 데이터를 유기적으로 연결하고, 데이터 기반 의사결정을 지원하는 통합 플랫폼으로 발전시켜 나갈 것입니다.

> **참고:** 본 문서는 과업 수행 계획 및 사전 설계 결과를 정리한 초안입니다. 실제 과업 수행 후 결과 데이터로 갱신될 예정입니다.

| 역할 기반 접근 제어 | Role-Based Access Control (RBAC) | 사용자 역할에 따라 데이터 접근 권한을 제어하는 방식 |

| 그래프 기반 RAG | GraphRAG | 그래프 구조를 활용한 검색 증강 생성 |
