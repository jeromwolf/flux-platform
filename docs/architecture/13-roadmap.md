# 13. 연차별 아키텍처 진화 (Y1-Y5)

[← 설계 원칙](./12-design-principles.md) | [다음: 디렉토리 구조 →](./14-directory-structure.md)

## 개요

IMSP 플랫폼은 5개년(2026-2030)에 걸쳐 점진적으로 확장되는 아키텍처를 채택한다. 1단계(Y1-Y3)에서는 핵심 기능 구축과 MVP 달성을, 2단계(Y4-Y5)에서는 고도화와 안정 운영을 목표로 한다. 본 문서는 컴포넌트별 진화 로드맵, 정량 목표, 마일스톤 타임라인, 위험 관리 전략을 기술한다.

---

## 13.1 컴포넌트별 진화 로드맵

| 컴포넌트 | Y1 (2026) | Y2 (2027) | Y3 (2028) | Y4 (2029) | Y5 (2030) |
|----------|-----------|-----------|-----------|-----------|-----------|
| **인증** | JWT → Keycloak PoC | Keycloak OIDC 전환 완료 | SSO 완성 + MFA | ISMS-P 인증 준비 | 안정 운영 |
| **워크플로우** | Activepieces (현행 유지) | VueFlow 캔버스 + Argo v1 | Argo DAG 완성 | 자동화 + 최적화 | 안정 운영 |
| **LLM 서빙** | CPU PoC (Ollama) | GPU (vLLM, 1x A100) | Ray 분산 클러스터 | Quantization 최적화 | SLA 관리 |
| **KG 규모** | 10K nodes, 50K rels | 100K nodes, 500K rels | 500K nodes, 2M rels | 1M nodes, 5M rels | 2M+ nodes, 10M+ rels |
| **온톨로지** | 40 types (재설계 완료) | 70 types (+S-100 매핑) | 100 types (통합) | 130 types (고도화) | 150 types (안정) |
| **동시 사용자** | 5명 | 10명 | 30명 | 50명 | 100+명 |
| **모니터링** | Prometheus + Grafana 기본 | AlertManager + EFK 도입 | 자동 스케일링 연동 | SLA 대시보드 | SLA 99.9% 관리 |
| **K8s** | dev/staging 클러스터 | prod 클러스터, Helm v1 | GPU 클러스터 통합 | HA (Multi-AZ 검토) | 자동 복구 |
| **데이터 처리** | Batch (CronJob) | Batch + Kafka 스트리밍 | 실시간 AIS 스트리밍 | Ray 분산 처리 | 글로벌 데이터 통합 |
| **보안** | API Key + JWT | Keycloak RBAC | 감사 로그 자동화 | ISMS-P | 보안 자동화 |
| **프론트엔드** | VueFlow PoC (Q3) | VueFlow 기본 캔버스 | 전자해도 + 3D 렌더러 | 대시보드 고도화 | 서비스 포털 완성 |
| **SLA** | 95% (내부) | 99% | 99.5% | 99.8% | 99.9% |

## 13.4 1차년도 상세 일정 (Y1: 2026)

1차년도는 플랫폼의 기반을 확립하는 핵심 시기이다. 아래 표는 분기별 주요 산출물과 마일스톤을 상세히 기술한다.

| 분기 | 주요 산출물 | 마일스톤 | 핵심 활동 |
|------|-----------|---------|----------|
| **Q1** (2026.01-03) | 운영 개념 정의서, KG 엔진 이식 완료, 아키텍처 상세 설계서 | **M1:** 아키텍처 설계 완료 | flux-n8n → flux-platform 코드 마이그레이션, 1,136 테스트 통과 확인, core/kg/ 모듈 구조 안정화, 5개년 전략서 작성 |
| **Q2** (2026.04-06) | 온톨로지 v1 (~40 엔티티), ELT 파이프라인 설계, 기술 벤치마크 보고서 | **M2:** 온톨로지 v1 확정 | KRISO 요구사항 기반 온톨로지 재설계 (기존 127 → 40 엔티티 선별), OWL/Turtle 파일 작성, S-100 매핑 사전 조사, LLM 벤치마크 (Qwen 2.5 vs Llama 3) |
| **Q3** (2026.07-09) | API v1 (REST), Keycloak PoC, VueFlow 캔버스 PoC, 노드 5종 정의, NLP/NER PoC | **M3:** API + 인증 통합 | FastAPI 라우트 구현 (query/graph/schema/etl/lineage/health), Keycloak realm 설정, VueFlow 캔버스 프로토타입, 워크플로우 노드 5종 스펙 정의, NLP/NER 엔티티 인식 파이프라인 PoC |
| **Q4** (2026.10-12) | Docker Compose 배포, 30문항 평가 데이터셋, 1차년도 보고서 | **M4:** Y1 납품 | 통합 PoC 데모 (KRISO 발표), Text2Cypher 70% 정확도 달성, 평가 프레임워크 구축, 1차년도 최종 보고서 |

### Q1 세부 타임라인

```
Week 01-02: 프로젝트 킥오프, 개발 환경 구축 (Git, CI/CD, Docker)
Week 03-04: flux-n8n 코드 분석 및 마이그레이션 계획 수립
Week 05-08: core/kg/ 이식 + 테스트 통과 확인
Week 09-10: domains/maritime/ 이식 + 테스트 통과 확인
Week 11-12: 아키텍처 상세 설계서 작성 + KRISO 중간 보고
```

## 13.5 위험 관리

### 13.5.1 위험 요소 분석

| 위험 요소 | 가능성 | 영향도 | 대응 전략 |
|----------|--------|--------|----------|
| Neo4j CE 성능 한계 도달 | 중 | 높음 | Y3 이전 EE 전환 예산 확보, 애플리케이션 캐싱 강화 |
| 온프레미스 GPU 조달 지연 | 중 | 중간 | Y1: CPU 기반 PoC, Y2 초 GPU 우선 확보 계획 |
| 한국어 해사 LLM 성능 미달 | 높음 | 높음 | Qwen 2.5 VL 파인튜닝 데이터셋 Y1부터 구축 시작 |
| Keycloak 전환 복잡도 | 중 | 중간 | 기존 JWT 코드와 병행 운영, 점진적 마이그레이션 |
| KRISO 요구사항 변경 | 높음 | 중간 | 플러그인 아키텍처로 도메인 모델 교체 용이하게 설계 |

### 13.5.2 위험 완화 전략 상세

각 주요 위험에 대한 구체적인 완화 전략을 기술한다.

#### Risk 1: 온톨로지 재설계 지연

- **문제:** 기존 127 엔티티를 폐기하고 KRISO 요구사항 기반으로 완전 재설계해야 하나, 요구사항 확정이 지연될 수 있음
- **완화 전략:**
  - 기존 127 엔티티 중 핵심 40개를 우선 선정하여 Y1 Q2까지 v1 확정
  - 3-Layer Ontology 구조(Conceptual → Mapping → Physical)를 활용하여 Conceptual만 먼저 확정
  - 나머지 엔티티는 Y2-Y3에 걸쳐 점진적 추가 (70 → 100 → 130 → 150)
  - OWL/Turtle 파일 기반으로 온톨로지 변경을 버전 관리
- **트리거 조건:** Y1 Q2 종료 시점에 40개 엔티티 미확정 시 → KRISO와 긴급 워크숍 개최

#### Risk 2: LLM 성능 부족

- **문제:** 온프레미스 LLM(CPU 기반, Y1)으로 해사 도메인 한국어 Text2Cypher 70% 정확도 달성이 어려울 수 있음
- **완화 전략:**
  - Few-shot Exemplar DB 구축으로 LLM 프롬프트 품질 강화
  - Rule-based fallback: Direct Path(CypherBuilder) 우선 라우팅, LLM은 복잡한 쿼리에만 사용
  - 평가 데이터셋 30문항을 난이도별 분류 (Easy 15 / Medium 10 / Hard 5)
  - Easy 쿼리(단순 매칭)는 Rule-based로 100% 처리, LLM은 Medium+Hard에 집중
- **트리거 조건:** Y1 Q3 평가에서 50% 미만 시 → GPU 조기 확보 요청 + 외부 LLM API 임시 활용 검토

#### Risk 3: Suredata Lab 연동 지연

- **문제:** Suredata Lab의 데이터 수집/마트/AI서빙/DW 파이프라인 개발 일정이 IMSP 일정과 불일치할 수 있음
- **완화 전략:**
  - Mock API 서버 구축으로 Suredata 미완성 상태에서도 개발 병행
  - 데이터 인터페이스 스펙(JSON Schema)을 Y1 Q1에 사전 합의
  - Suredata → IMSP 데이터 경로: `Suredata DW → Object Storage (Ceph) → ETLPipeline → Neo4j`
  - Object Storage를 버퍼로 활용하여 양측 일정 디커플링
- **트리거 조건:** Y1 Q2까지 인터페이스 스펙 미합의 시 → PM 에스컬레이션

#### Risk 4: K8s 인프라 미확보

- **문제:** KRISO 온프레미스 K8s 클러스터 구축이 Y1 내에 완료되지 않을 수 있음
- **완화 전략:**
  - Y1은 Docker Compose 기반 로컬/테스트 환경으로 개발 진행
  - `infra/docker-compose.yml`에 Neo4j + Redis + Ceph + Keycloak 전체 스택 정의
  - Helm 차트는 Y1 Q3부터 사전 준비, K8s 클러스터 확보 즉시 배포 가능하도록 설계
  - Y2 Q1에 K8s 전환을 목표로 하되, Docker Compose 환경도 유지하여 이중 배포 지원
- **트리거 조건:** Y1 Q4까지 K8s 클러스터 미확보 시 → 클라우드 임시 환경 검토 (비용 별도)
