## 1장. 개요

### 1.1 목적

본 문서는 KRISO 대화형 해사서비스 플랫폼의 품질 보증을 위한 테스트 계획을 정의한다.
해사 도메인 지식그래프(Knowledge Graph)를 핵심으로 하는 본 플랫폼의 각 모듈이
설계 명세에 부합하며 안정적으로 동작함을 검증하기 위한 체계적인 테스트 전략,
대상 모듈별 테스트 케이스, 자동화 도구 구성, 커버리지 목표 등을 포함한다.

### 1.2 범위

1차년도 PoC 범위 내에서 다음 모듈을 테스트 대상으로 한다:

|------|------|------|
| 핵심 | `kg/ontology/core.py` | Ontology, ObjectType, LinkType 정의 클래스 |

## 2장. 테스트 전략

### 2.1 테스트 레벨

본 프로젝트의 테스트는 4단계 레벨로 구성한다.

#### 2.1.1 단위 테스트 (Unit Test)

- **목적**: 개별 모듈/클래스/함수의 독립적 기능 검증
- **대상**: Ontology Core, CypherBuilder, QueryGenerator, RBAC Models
- **특징**: 외부 의존성(Neo4j, 네트워크) 없이 순수 Python 로직 검증
- **마커**: `@pytest.mark.unit`
- **실행 빈도**: 매 커밋 시 CI/CD에서 자동 실행

#### 2.1.2 통합 테스트 (Integration Test)

- **목적**: Neo4j 데이터베이스와의 연동 검증
- **대상**: 스키마 초기화, 데이터 CRUD, 쿼리 실행, RBAC 정책 실행
- **특징**: Docker Compose 기반 Neo4j 컨테이너 필요
- **마커**: `@pytest.mark.integration`
- **실행 빈도**: PR 머지 전, 일일 CI/CD 빌드

#### 2.1.3 E2E 테스트 (End-to-End Test)

- **목적**: 시나리오 기반 종단간 검증 (자연어 질문 -> Cypher -> Neo4j -> 결과)
- **대상**: NL 쿼리 시나리오 5종 (공간, 관계, 시설, 사고, 기상)
- **특징**: 전체 파이프라인 동작 검증
- **마커**: `@pytest.mark.e2e`
- **실행 빈도**: 릴리즈 전, 주간 회귀 테스트

#### 2.1.4 성능 테스트 (Performance Test)

- **목적**: 벤치마크 기준 설정 및 성능 회귀 감지
- **대상**: Neo4j 쿼리 실행, CypherBuilder 구성, QueryGenerator 생성, Ontology 로딩
- **특징**: 반복 실행 후 통계(avg, p50, p95) 측정
- **마커**: `@pytest.mark.slow`
- **실행 빈도**: 릴리즈 전 수동 실행

### 2.2 테스트 설계 원칙

1. **독립성**: 각 테스트는 다른 테스트에 의존하지 않고 독립적으로 실행 가능해야 한다.
2. **반복성**: 동일 환경에서 동일 결과를 재현할 수 있어야 한다.
3. **가독성**: 테스트 코드는 명확한 의도를 전달하고, 실패 시 원인을 바로 파악할 수 있어야 한다.
4. **빠른 피드백**: 단위 테스트는 1초 이내, 통합 테스트는 30초 이내에 완료되어야 한다.
5. **최소 의존성**: 가능한 한 외부 서비스 없이 동작하도록 설계한다.

### 2.3 테스트 데이터 전략

| 계층 | 데이터 소스 | 관리 방식 |
|------|-------------|-----------|
| 단위 테스트 | 테스트 코드 내 인라인 fixture | pytest fixture, dataclass 인스턴스 |
| 통합 테스트 | conftest.py의 Neo4j fixture | 세션별 트랜잭션 롤백 |
| E2E 테스트 | `kg.schema.load_sample_data` | Docker Compose + 샘플 데이터 |
| 성능 테스트 | 샘플 데이터 기반 | 별도 벤치마크 스크립트 |

## 4장. 자동화 도구

### 4.1 테스트 프레임워크

| 도구 | 버전 | 용도 |
|------|------|------|
| pytest | 7.x+ | 테스트 실행, 마커, fixture, parametrize |
| pytest-cov | 4.x+ | 코드 커버리지 측정 |
| pytest-timeout | 2.x+ | 테스트 타임아웃 관리 |

### 4.2 pytest 마커 구성

```ini
[pytest]
markers =
    unit: 순수 Python 단위 테스트 (외부 의존성 없음)
    integration: Neo4j 연결 필요 통합 테스트
    e2e: 종단간 시나리오 테스트
    slow: 장시간 실행 테스트 (벤치마크 등)
```

### 4.3 마커별 실행 명령

```bash
# 단위 테스트만 실행 (CI 기본)
pytest -m unit -v

# 통합 테스트 실행 (Neo4j 필요)
pytest -m integration -v

# E2E 테스트 실행
pytest -m e2e -v

# 전체 실행 (단위 + 통합 + E2E)
pytest -v

# 커버리지 포함 실행
pytest -m unit --cov=kg --cov-report=html --cov-report=term-missing

# 느린 테스트 제외
pytest -m "not slow" -v
```

### 4.4 conftest.py 구조

```
tests/
  conftest.py           # 공통 fixture 정의
    - neo4j_driver       : module-scoped Neo4j 드라이버
    - neo4j_session      : function-scoped 세션 (rollback)
    - sample_ontology    : 미리 로딩된 해사 온톨로지
  test_ontology_core.py  # Ontology Core 단위 테스트
  test_maritime_ontology.py  # Maritime Ontology 단위 테스트
  test_cypher_builder.py     # CypherBuilder 단위 테스트
```
