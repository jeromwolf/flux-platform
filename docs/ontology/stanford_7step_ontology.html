1.1 온톨로지 설계 목적

대화형 해사서비스 플랫폼(IMSP)의 자원 의미 연결을 위한 지식그래프 온톨로지를 설계한다.

#목적설명
P1멀티모달 해사 데이터 통합AIS, GPS, RADAR, VHF, Port-MIS, CCTV, 기상, 시험시설 등 이질적 데이터를 하나의 의미 체계로 연결
P2자연어 기반 지식 탐색Text-to-Cypher 파이프라인을 통해 비전문가도 자연어로 KG를 질의할 수 있도록 지원
P3KRISO 시험시설 데이터 구조화선형수조, 캐비테이션터널 등 시험시설의 시험 이력/결과/보고서를 체계적으로 관리
P4국제 표준 정합성 확보IHO S-100, IMO AIS 표준 기반 온톨로지로 국제 해사 데이터와의 상호운용성 보장
P52차년도 LLM 연계 기반 마련GraphRAG + LLM 대화형 인터페이스와 즉시 연계 가능한 스키마 설계

LPG 기반

Neo4j Community Edition 사용 (Labeled Property Graph). RDF 대비 직관적 쿼리 성능 우위.

Schema-optional

유연한 확장성 + SHACL 사전 품질 검증으로 무결성 보장. 스키마 강제 없이 품질 확보.

Cypher 최적화

Text-to-Cypher 변환 시 직관적인 노드/관계 명명 규칙 적용. PascalCase 노드, SCREAMING_SNAKE 관계.

URN 식별 체계

urn:kriso:kg:{type}:{subtype}:{id} 형식의 글로벌 고유 식별자.

1.4 Competency Questions (CQ) — 핵심 5개

Stanford 7-Step에서 CQ는 온톨로지가 반드시 답할 수 있어야 하는 질문이다. 이 질문들이 Step 3(핵심 용어), Step 4(클래스 계층), Step 5(속성)를 이끈다.

🔎 단계별 CQ 체계 (Step-level CQ Framework)

마스터 CQ(CQ1~CQ5)는 온톨로지의 최종 목표를 정의한다. 각 단계는 자기 산출물이 마스터 CQ를 향해 올바르게 진행되고 있는지 검증하는 단계 CQ(Step-CQ)를 가진다.

Step산출물단계 CQ 검증 질문
Step 1목적 · 범위 · 마스터CQCQ1~CQ5 정의 (이 섹션) ✓
Step 2재사용 분석S2-CQ1~3: 재사용 요소가 CQ를 커버하는가?

1-hop · Direct
CQ1: "VLCC 모형선의 14노트 저항계수(Ct)는?"
관련 엔티티
Vessel TestScenario MeasurementRecord
필요 관계
Vessel -[TESTED_IN]-> TestScenario -[PRODUCES]-> MeasurementRecord
MATCH (v:Vessel {vesselType: "VLCC"})-[:TESTED_IN]->(ts:TestScenario)
      -[:PRODUCES]->(mr:MeasurementRecord {speed: 14})
RETURN mr.resistanceCoefficient AS Ct
2-hop · Bridge
CQ2: "설계 EHP(유효마력)와 실해역 연시(Sea Trial) 결과의 차이는?"
관련 엔티티
Vessel TestScenario MeasurementRecord VoyageEvent
추론 유형
Bridge (순차 이동 A→B→C)
MATCH (v:Vessel)-[:TESTED_IN]->(ts:TestScenario)-[:PRODUCES]->(mr:MeasurementRecord)
MATCH (v)-[:PARTICIPATED_IN]->(ve:VoyageEvent)
WHERE mr.parameterType = "EHP" AND ve.hasPerformanceData = true
RETURN mr.value AS designEHP, ve.actualEHP AS seaTrialEHP,
       abs(mr.value - ve.actualEHP) AS difference
2-hop · Bridge
CQ3: "부산항 VTS 관제구역에 적용되는 COLREGS(국제해상충돌예방규칙) 조문은?"
관련 엔티티
Port SeaArea Regulation
필요 관계
Port -[HAS_VTS_ZONE]-> SeaArea <-[APPLIES_TO]- Regulation
MATCH (p:Port {name: "부산항"})-[:HAS_VTS_ZONE]->(sa:SeaArea)
MATCH (r:Regulation)-[:APPLIES_TO]->(sa)
WHERE r.framework = "COLREGS"
RETURN r.articleNumber, r.title, sa.name
2-hop · Composition
CQ4: "파고 3m 이상 조건에서 선박 저항 증가율은?"
관련 엔티티
SeaArea OceanEnvironment MeasurementRecord TestCondition
추론 유형
Composition (환경 조건 + 성능 데이터 결합)
MATCH (tc:TestCondition)-[:PRODUCES]->(mr:MeasurementRecord)
WHERE tc.waveHeight >= 3.0
RETURN tc.waveHeight, mr.resistanceCoefficient,
       mr.resistanceIncrease AS increaseRate
ORDER BY tc.waveHeight
Multi-hop · Intersection
CQ5: "캐비테이션터널에서 프로펠러 공동 조건이 유사한 시험 목록은?"
관련 엔티티
TestFacility TestScenario TestCondition Sensor
필요 관계
TestFacility -[HOSTS]-> TestScenario -[APPLIES]-> TestCondition
MATCH (tf:TestFacility {facilityType: "캐비테이션터널"})-[:HOSTS]->(ts:TestScenario)
      -[:APPLIES]->(tc:TestCondition)
WHERE tc.cavitationType IS NOT NULL
WITH tc.pressure AS pressure, tc.propellerRPM AS rpm, collect(ts) AS scenarios
WHERE size(scenarios) > 1
RETURN pressure, rpm, size(scenarios) AS similarCount,
       [s IN scenarios | s.scenarioId] AS scenarioIds

S3-CQ2

검증 대상: P0/P1 경계 검증 — P0만으로 답할 수 없는 CQ가 있다면 해당 엔티티를 P0로 승격해야 하는지 판단
S3-CQ3
신규 엔티티 4개(VoyageEvent, AISRecord, TestScenario, MeasurementRecord)가 없으면 어떤 CQ가 깨지는가?
검증 대상: 신규 엔티티 필요성 — 4개 NEW 엔티티 각각이 커버하는 CQ를 역추적하여 추가 근거 확인
