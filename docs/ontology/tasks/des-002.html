DES-002 · 지식그래프 물리 스키마 설계

선정된 그래프 DB의 기술적 특성을 반영하여 대규모 해사 데이터를 효율적으로 저장·탐색할 수 있는 물리적 데이터 구조를 설계한다.

1

목적

DES-001이 논리 온톨로지라면 DES-002는 그것을 실제 Neo4j에 어떻게 저장할지 설계한다. 노드 레이블·관계 타입·속성 키 명명 규칙, 시공간 인덱스, 멀티모달 속성 수용 구조, 스키마 버전 관리가 핵심이다.

2

수행 방법

그래프 DB 물리 모델링·최적화

노드 레이블·관계 타입·속성 키 명명 규칙 + 질의 성능 위한 정규화/비정규화 전략.

시공간·대용량 인덱스 설계

선박 위치·궤적용 공간 인덱스(point) + 시간축 인덱스 + 빈번 속성 전역 인덱스·제약조건.

멀티모달 데이터 속성 수용 구조

정형(AIS)·비정형(법령)·시공간 데이터 메타데이터 저장 구조 + 대용량 바이너리는 외부 저장소 하이브리드.

스키마 버전 관리·파티셔닝

2차년도 확장 대비 마이그레이션 체계 + 노드/엣지 파티셔닝·샤딩 전략 + Batch Size 정의.

3

문제점 도출

3건
HIGHISSUE-01 보유 docs/DES-002는 "멀티모달 통합 설계서" — KRISO DES-002는 "물리 스키마 설계" 미해결
현상
파일명·내용이 KRISO 요구와 불일치. 보유 문서를 그대로 제출하면 검수 부적합.
대응 방안
보유 DES-002 문서의 멀티모달 부분은 REQ-001/DES-003으로 분산 활용하고, 물리 스키마 설계서는 신규 작성.
MIDISSUE-02 DES-001 온톨로지 확정 전 물리 스키마 미확정 (선행 의존) 진행 중
현상
물리 스키마는 논리 온톨로지(DES-001)를 입력으로 한다. DES-001 Step 5-7 완료 후 착수 가능.
대응 방안
DES-001과 동일 마감(M+5). DES-001 Step 4(클래스 계층)까지 확정분으로 명명 규칙·인덱스 설계 선행.
LOWISSUE-03 공간 인덱스 — Neo4j point() vs WKT 선택 미해결
현상
Neo4j 네이티브 point()는 간단하나 복잡한 공간 연산 한계. WKT+공간 플러그인은 강력하나 복잡.
대응 방안
PoC 범위는 point() 우선, 복잡 공간 연산 필요 시 2차년도 검토 사항으로 명시.
