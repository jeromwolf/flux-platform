1

목적

DES-004는 "만들어진 KG를 어떻게 쓸까"를 설계한다. 최단경로·다단계 탐색 알고리즘, 해사 서비스 복합 질의 시나리오, GraphQL/REST API, 그리고 2차년도 LLM/GraphRAG 연계를 고려한 모듈형 아키텍처가 핵심이다.

2

수행 방법

지식 탐색·경로 분석 알고리즘

최단경로(Shortest Path) · 가변길이 경로(1~3-hop) · 중심성(Centrality) · 유사도 분석.

해사 특화 복합 질의 시나리오

"사고 발생 시 적용 법령", "선박-시험시설 연계 분석" 등 도메인 시나리오 + 시공간·속성 결합 쿼리 템플릿.

API·인터페이스 아키텍처

GraphQL(유연한 그래프 질의) + REST(외부 연동) 이중 인터페이스 + 2차년도 LLM 연계 모듈형 설계.

시각화 UI/UX·쿼리 최적화

그래프 시각화 레이아웃·필터링 UI + 인덱스 튜닝·결과 캐싱(TTL) 전략.

3

문제점 도출

3건
MIDISSUE-01 2차년도 LLM/GraphRAG 연계 — 1차년도 설계 범위 경계 미해결
현상
자연어→Cypher 변환·GraphRAG는 2차년도 구현 대상. 1차년도 설계에 어디까지 포함할지 모호.
대응 방안
1차년도는 "모듈형 아키텍처 설계"까지 — 자연어→Cypher 모듈·그래프 컨텍스트 추출 모듈을 인터페이스 수준으로 정의, 구현은 2차년도로 명시.
LOWISSUE-02 GraphQL vs REST 우선순위 미해결
현상
이중 인터페이스를 제안했으나 1차년도 PoC에서 우선 검증할 인터페이스 선택 필요.
대응 방안
PoC(TER)는 REST 우선 검증, GraphQL은 설계 명세로 제공.
LOWISSUE-03 쿼리 응답 성능 목표의 설계 반영 미해결
현상
복합 쿼리 5초 이내 목표를 설계 단계에서 보장할 인덱스·캐싱 전략이 필요.
대응 방안
DES-002 인덱스 설계와 연계하여 빈번 복합 질의용 인덱스·캐싱(TTL)을 아키텍처에 명시.
