1. **import 경로**: `pyproject.toml`의 `include = ["kg*", "maritime*"]`와 물리 디렉토리 `core/kg/`, `domains/maritime/` 불일치
2. **중복 코드**: `core/kg/crawlers/` ↔ `domains/maritime/crawlers/` 등 중복
3. **인증**: JWT 기반 → Keycloak OIDC 전환 필요
4. **워크플로우**: Activepieces → Argo Workflow 전환 필요
5. **ETL**: ETL → ELT 패러다임 전환 필요

## 5. 온톨로지 재설계 방향 (v1 목표 ~40 엔티티)

### 설계 원칙
1. **Data-Driven**: 확인된 데이터 소스가 있는 엔티티만 정의
2. **Maritime Traffic First**: AIS 기반 해상교통 도메인 우선
3. **Temporal**: 모든 규칙/제약에 validFrom/validTo 적용
4. **ELT-Friendly**: Raw → Metadata → Entity 3단계 리니지 추적
5. **S-100 Compatible**: IHO S-100 표준 엔티티와 1:1 매핑
6. **Multi-Layer**: 정적 지식 / 동적 관측 / 파생 인사이트 분리

### 목표 구조
```
IMSP Ontology v1 (Maritime Traffic)
├── StaticKnowledge (~12 types)
│   ├── Port, Berth, Terminal, Waterway
│   ├── TSS, Regulation, TrafficRule
│   ├── RestrictedArea, TemporalConstraint
│   └── SeaArea, ChartFeature
├── DynamicObservation (~10 types)
│   ├── Vessel, VesselPosition, TrackSegment
│   ├── AISMessage, WeatherObservation
│   ├── OceanCurrent, RadarContact
│   └── PortCall
├── DerivedInsight (~6 types)
│   ├── RouteDeviation, SpeedAnomaly
│   ├── CollisionRisk, TrafficDensity
│   └── AnomalyEvent
└── DataLineage (~4 types)
    ├── RawAsset, ProcessedAsset
    ├── TransformActivity
    └── ProvenanceChain
```

## 6. Suredata Lab 연동 API (v1)

```
# Suredata → flux-platform (데이터 주입)
POST /api/v1/ingest/raw           # 원천 데이터 등록
POST /api/v1/ingest/metadata      # 메타데이터 등록
POST /api/v1/ingest/entities      # 추출 엔티티 등록

# flux-platform → Suredata (결과 조회)
GET  /api/v1/query/nl             # 자연어 쿼리
GET  /api/v1/query/cypher         # Cypher 직접 쿼리
GET  /api/v1/lineage/{asset_id}   # 리니지 조회
GET  /api/v1/ontology/schema      # 온톨로지 스키마 조회
```
