### 1.2 과업 수행 배경

본 과업은 한국해양과학기술원 부설 **선박해양플랜트연구소(KRISO)**의 해사 데이터를 중심으로, 산재된 해사 도메인 데이터를 **지식그래프(Knowledge Graph)**로 통합하고 **대화형 AI 인터페이스**를 통해 비전문가도 해사 데이터에 접근할 수 있는 차세대 서비스 플랫폼을 구축하는 것을 목표로 한다.

1차년도는 **"설계 + PoC"** 단계로, 해사 도메인 온톨로지 설계, 지식그래프 스키마 구축, ETL 파이프라인 프레임워크 개발, 그리고 GraphRAG 기반 자연어 질의응답 프로토타입 검증까지를 수행한다.

## 2. 연구 배경 및 필요성

### 2.1 해사 데이터 사일로 문제

현재 국내 해사 분야의 데이터는 **다수의 기관에 분산**되어 관리되고 있으며, 기관 간 데이터 상호운용성이 매우 낮은 상황이다.

| 기관 | 보유 데이터 | 현재 관리 방식 | 문제점 |
|------|-----------|---------------|--------|
| 해양수산부 | 해사 정책, 규정, 공시 데이터 | 별도 DB + 문서 | 비정형 데이터 검색 난이 |
| 국립해양조사원 | 해도, 조석, 해류, 수심 | 전용 시스템 | API 표준화 미흡 |
| 기상청 | 해양 기상, 파고, 풍속, 수온 | 실시간 스트림 | KG 미연계 |
| 해양경찰청 | 해양 사고, 수색 구조, 불법 활동 | 내부 시스템 | 타 기관 데이터 연계 부재 |
| KRISO | 선박 모형 시험, 실험 결과, 연구 논문 | 개별 파일 시스템 | 메타데이터 체계 부재 |

**핵심 문제**: 해사 공무원, 어민, 해운사, 연구자 등 다양한 이해관계자가 해사 데이터에 접근하려면 **각 기관의 개별 시스템을 파악**하고 **전문 쿼리 언어(SQL 등)**를 사용해야 하며, **기관 간 데이터를 수동으로 매칭**해야 한다.

### 2.2 지식그래프 기반 통합의 필요성

지식그래프(Knowledge Graph)는 이질적인 데이터를 **엔티티(Entity)**와 **관계(Relationship)**의 그래프 구조로 통합하여, 다음의 핵심 가치를 제공한다:

1. **크로스도메인 연결**: 선박-항만-화물-사고-기상-규정 간 관계를 명시적으로 표현
2. **멀티홉 추론**: "A 선박이 적재한 위험물 → 해당 규정 → 최근 위반 사례"와 같은 다단계 추론
3. **자연어 접근**: Text-to-Cypher 변환을 통해 비전문가도 자연어로 데이터 조회 가능
4. **벡터 검색 통합**: 그래프 구조 + 벡터 임베딩의 결합으로 의미 기반 검색 지원
5. **온톨로지 기반 품질 관리**: OWL 표준 온톨로지로 데이터 일관성 자동 검증

## 4. 연구 내용 및 방법

### 4.1 해사 도메인 온톨로지 설계

#### 4.1.1 설계 방법론

**"W3C 표준 기반 설계, Neo4j로 구현"** 원칙을 채택한다.

```
┌──────────────────┐     ┌───────────────────┐     ┌──────────────────┐
│   Protege IDE    │────>│  OWL 2 Turtle     │────>│  Neo4j Property  │
│  (온톨로지 설계)  │     │  (maritime.ttl)   │     │  Graph (n10s)    │
└──────────────────┘     └───────────────────┘     └──────────────────┘
```

### A.6 지식그래프·AI 관련 용어

| 용어 | 영문 | 설명 |
|------|------|------|
| 지식그래프 | Knowledge Graph (KG) | 데이터를 '노드(사물)'와 '엣지(관계)'로 표현하는 그래프 데이터베이스. "부산항 ←DOCKED_AT← 해운대호" 처럼 관계를 명시 |
| 온톨로지 | Ontology | 특정 분야의 개념과 관계를 체계적으로 정의한 '사전'. 기계가 읽을 수 있는 분류 체계 |
| OWL | Web Ontology Language | W3C가 정한 온톨로지 표현 표준 언어 |
| Turtle (TTL) | Terse RDF Triple Language | RDF 데이터를 간결하게 작성하는 형식. OWL 온톨로지를 저장하는 파일 형식 |
| Cypher | - | Neo4j 그래프 데이터베이스의 쿼리 언어. SQL의 그래프 버전 |
| GraphRAG | Graph Retrieval-Augmented Generation | 지식그래프에서 관련 정보를 검색(Retrieval)하여 LLM의 답변을 보강(Augmentation)하는 기법 |
| Text-to-Cypher | - | 자연어 질문을 Cypher 쿼리로 자동 변환하는 기술. "부산항 선박 보여줘" → `MATCH (v:Vessel)-[:DOCKED_AT]->(p:Port {name:'부산항'}) RETURN v` |
| 임베딩 | Embedding | 텍스트/이미지를 AI가 비교할 수 있는 숫자 벡터(768차원)로 변환한 것. '의미의 좌표' |
| 환각 | Hallucination | AI가 사실이 아닌 정보를 그럴듯하게 생성하는 현상. 온톨로지 기반 검증으로 방지 |
| PROV-O | Provenance Ontology | W3C 표준 데이터 출처(리니지) 추적 온톨로지 |
| RBAC | Role-Based Access Control | 역할 기반 접근 제어. 사용자의 역할(관리자, 연구자 등)에 따라 접근 가능한 데이터를 제한 |
| ETL | Extract, Transform, Load | 데이터를 추출→변환→적재하는 파이프라인 |

### A.7 데이터 형식·표준

| 용어 | 영문 | 설명 |
|------|------|------|
| IHO S-100 | Universal Hydrographic Data Model | 국제수로기구의 해양 데이터 표준 프레임워크. 전자해도(ENC) 등의 기반 |
| n10s (Neosemantics) | - | Neo4j에서 OWL/RDF 데이터를 처리하는 플러그인. OWL 온톨로지를 그래프로 자동 변환 |
| APOC | Awesome Procedures on Cypher | Neo4j의 확장 프로시저 라이브러리. 데이터 변환, 가져오기 등 유틸리티 모음 |
| WGS84 | World Geodetic System 1984 | 전 세계 GPS가 사용하는 좌표계 (위도/경도) |
| HNSW | Hierarchical Navigable Small World | 벡터 유사도 검색에 사용되는 고속 인덱스 알고리즘 |

## 부록 B. 온톨로지 클래스·속성 상세

본 프로젝트의 해사 도메인 온톨로지(127 엔티티, 83 관계, 29 속성정의)를 10개 엔티티 그룹별로 상세히 기술한다. 각 클래스의 의미, 하위 유형, 주요 속성을 포함하며, 실제 구현 코드(`kg/ontology/maritime_ontology.py`)에 기반한다.

### B.1 PhysicalEntity — 물리적 객체

물리적으로 존재하는 해사 객체. 바다 위에서 움직이거나, 항만에 고정되어 있는 실체.

| grossTonnage | FLOAT | | 총톤수 (GT) |
| deadweight | FLOAT | | 재화중량톤수 (DWT) |
| length | FLOAT | | 전장 (m) |
| beam | FLOAT | | 선폭 (m) |
| draft | FLOAT | | 흘수 (m) |
