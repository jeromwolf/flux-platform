## 1. Executive Summary

### 1.1 프로젝트 비전

IMSP는 해사(Maritime) 도메인의 이질적 데이터(AIS 궤적, 레이더, CCTV, 전자해도, 기상, 법령 문서 등)를 **지식그래프(Knowledge Graph)** 로 통합하고, **자연어 대화** 만으로 데이터 탐색, 워크플로우 생성, 도메인 추론을 수행할 수 있는 **대화형 해사서비스 설계 플랫폼**이다.

5개년에 걸쳐 "설계 + 개념검증(1차년도)" → "구성요소 개발(2차년도)" → "MVP 통합(3차년도)" → "고도화(4차년도)" → "안정화 + 민간 지원(5차년도)" 순으로 진행하며, 최종적으로 **29건의 민간 지원**, **SCI 4편 + KCI 4편**, **특허 8건 출원/6건 등록**을 달성한다.

### 1.2 인사이트마이닝의 핵심 역할

인사이트마이닝은 IMSP 플랫폼의 **지식 계층(Knowledge Layer)** 전체와 **플랫폼 운영 인프라**를 책임진다.

| 영역 | 구체적 범위 |
|------|------------|
| **지식그래프 모델** | 온톨로지 설계(OWL + Neo4j), KG 구축/탐색/추론, GraphRAG |
| **자연어 탐색** | Text2Cypher 파이프라인, 한국어 NLP, 환각 감지, 품질 게이트 |
| **ELT 파이프라인** | Raw 데이터 → Object Storage → 메타데이터 추출 → KG 적재 |
| **리니지 추적** | W3C PROV-O 기반, 시간조건부 유효성(validFrom/validTo) |
| **도메인 모델** | 해사 특화 모델 5종/년 (총 25종), 해상교통 → 해양 전체 확장 |
| **워크플로우 노드** | 6종 노드 분류 체계 + VueFlow 기반 저작도구 |
| **가시화/상호작용** | 전자해도, 3D 렌더러, 대시보드, 에이전트 대화창 |
| **플랫폼 운영** | K8s 인프라, Keycloak 인증, 모니터링, 멀티테넌트 |
| **OCR/콘텐츠 분석** | 해사 문서 OCR, AI 엔티티 추출 |
| **민간 지원** | 총 29건 (1차 3건 ~ 5차 10건) |

### 1.3 차별화 요소

1. **기존 자산 활용**: flux-n8n의 KG 엔진(1,148 단위테스트, 성숙도 ★★★★★)을 코어로 재사용
2. **도메인 독립 설계**: `kg/`(엔진) + `maritime/`(플러그인) 분리 아키텍처 → 타 도메인 확장 용이
3. **시간조건부 지식 추론**: "3월에만 통항 금지인 구역을 4월에 추론" 같은 temporal reasoning 지원
4. **ELT 철학**: "일단 올려라" — Raw 데이터 우선 적재 후 AI 기반 엔티티 추출
5. **Agentic AI**: Orchestrator Agent → A2A Protocol → 하위 Agent Engine → MCP → Agent Tools

### 2.3 6대 개발 축 정의

IMSP 플랫폼은 다음 6대 개발 축으로 구성된다. 각 축은 5개년에 걸쳐 점진적으로 성숙해진다.

| # | 개발 축 | 핵심 내용 | 인사이트마이닝 담당 비중 |
|---|---------|----------|----------------------|
| 1 | **플랫폼 운영 기술** | K8s 인프라, 보안/멀티테넌트, KG 구축/탐색, 리니지 | **90%** (인프라 + KG 전체) |
| 2 | **통합개발환경 (IDE)** | VueFlow 워크플로우 저작도구, 노드 개발, 자동 생성 | **80%** (워크플로우 엔진 + UI) |
| 3 | **자원 저장소** | Object Storage + 서빙 데이터 저장소, 표준 인터페이스 | **40%** (KG 적재 파트, Suredata가 DW 담당) |
| 4 | **가시화/상호작용** | 전자해도, 3D 렌더러, 대시보드, 에이전트 대화 | **100%** |
| 5 | **도메인 모델** | 기본+고급 해사 모델 (매년 5종) | **100%** |
| 6 | **민간 지원** | 총 29건 (1차 3건 ~ 5차 10건) | **100%** |

### 2.4 핵심 용어 체계

IMSP 플랫폼 내에서 사용하는 용어의 정의를 통일한다.

| 용어 | 영문 | 정의 | 예시 |
|------|------|------|------|
| **자원 (Resource)** | Resource | 원천 데이터, 소스 코드, 서버 등 물리/논리적 개체 | AIS 수신기, 기상 API, GPU 서버 |
| **자산 (Asset)** | Asset | 플랫폼에 등록되어 메타데이터가 관리되는 모든 유무형 자산 | 등록된 데이터셋, AI 모델, 워크플로우 |
| **응용 (App)** | Application | 워크플로우 내에서 호출되는 독립적 실행 프로그램/컨테이너 | 선박 탐지 모델, OCR 엔진, 궤적 분석기 |
| **컴포넌트 (Component)** | Component | 특정 기능을 수행하는 노드들의 논리적 묶음 | AIS 수집 컴포넌트, 기상 분석 컴포넌트 |
| **어댑터 (Adapter)** | Adapter | 외부 데이터/시스템 연동을 위한 전용 컴포넌트 | AIS NMEA 어댑터, S-100 해도 어댑터 |
| **워크플로우 (Workflow)** | Workflow | 노드들을 연결하여 구성한 비즈니스 로직 | "AIS 수집 → 이상행동 탐지 → 알림" 워크플로우 |
| **서비스 (Service)** | Service | 완성된 워크플로우를 통해 사용자에게 제공되는 최종 기능 | 항만 혼잡도 모니터링 서비스 |
| **응용 vs 서비스** | App vs Service | 응용은 1회성 실행 + 결과 반환, 서비스는 지속 운용 | 응용: 궤적 분석 실행 / 서비스: 실시간 모니터링 |

### 2.5 사용자 레이어

IMSP는 3개의 사용자 레이어를 분리하여 보안과 접근 제어를 계층화한다.

```
┌─────────────────────────────────────────────────────────┐
│ Layer 0: 데이터 제공처 (Data Providers)                    │
│ ────────────────────────────────────────────────────     │
│ 해수부 │ 기상청 │ 해경 │ 자율운항센터 │ 국토지리원          │
│ → Raw 데이터 Push (API/파일/스트림)                       │
│ → 메타데이터 자동 추출 + KG 적재                           │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│ Layer 1: 1차 사용자 (Internal Researchers)                │
│ ────────────────────────────────────────────────────     │
│ KRISO 내부 연구자                                         │
│ → VueFlow 워크플로우 캔버스 (노드 편집)                     │
│ → 전역 대화창 (워크플로우 자동 생성, 커넥터 탐색)             │
│ → 노드별 대화창 (파라미터 설정, 디버깅, 실행 결과 확인)       │
│ → 팀/프로젝트 단위 환경 격리                                │
│ → Keycloak OIDC 인증 + RBAC                              │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│ Layer 2: 2차 사용자 (Public/External)                     │
│ ────────────────────────────────────────────────────     │
│ 해양공무원, 해운사, 어민, 해양레저, 해사연구자               │
│ → 서비스 포털 (읽기 전용, 완성된 서비스 소비)                │
│ → 과도한 사용 시 경고/정지                                 │
│ → 민간 지원 프로그램 (총 29건)                             │
└─────────────────────────────────────────────────────────┘
```

**접근 제어 정책:**

| 레이어 | 인증 | 권한 | 환경 격리 |
|--------|------|------|----------|
| Layer 0 (데이터 제공처) | API Key + IP 화이트리스트 | 쓰기 전용 (Ingest API) | 기관별 버킷/네임스페이스 |
| Layer 1 (내부 연구자) | Keycloak OIDC + MFA | 읽기/쓰기/실행 (RBAC) | 팀/프로젝트 단위 K8s Namespace |
| Layer 2 (2차 사용자) | Keycloak OIDC | 읽기 전용 (서비스 포털) | 공유 환경 + Rate Limit |

### 3.3 데이터 흐름 (ELT 파이프라인)

KRISO 미팅에서 확정된 **ELT(Extract-Load-Transform)** 철학: "일단 올려라, 올리고 나면 우리가 처리하겠다."

```
Phase 1: 수집 (Extract)
──────────────────────
AIS 수신기 ─┐
LTE 기지국  ─┤
레이더      ─┤
CCTV       ─┤    ┌──────────────────────┐
GPS/라이다  ─┼───→│ 수집 어댑터 (Adapter)  │
전자해도    ─┤    │ NMEA/GRIB2/S-100/    │
기상 API   ─┤    │ RTSP/REST/Kafka      │
법령 문서   ─┘    └──────────┬───────────┘
                             │
Phase 2: 적재 (Load)          ▼
──────────────────    ┌──────────────────────┐
                      │ Object Storage (MinIO)│
                      │ Raw 데이터 원본 보존    │
                      │ 버킷: raw/{source}/    │
                      │        {yyyy-mm-dd}/   │
                      └──────────┬───────────┘
                                 │
Phase 3: 메타데이터 추출           ▼
──────────────────────  ┌──────────────────────┐
                        │ 메타데이터 자동 추출    │
                        │ - EXIF (사진)          │
                        │ - GPS 좌표             │
                        │ - 파일 속성 (크기/형식)  │
                        │ - 타임스탬프             │
                        └──────────┬───────────┘
                                   │
Phase 4: 콘텐츠 분석 + KG 적재      ▼
──────────────────────────  ┌──────────────────────┐
                            │ AI 콘텐츠 분석         │
                            │ - OCR (PaddleOCR)     │
                            │ - 엔티티 추출 (NER)    │
                            │ - 관계 추출            │
                            │ - 임베딩 생성           │
                            └──────────┬───────────┘
                                       │
                                       ▼
                            ┌──────────────────────┐
                            │ Neo4j (KG)            │
                            │ 엔티티 + 관계 + 속성   │
                            │ + 벡터 인덱스           │
                            │ + 공간 인덱스           │
                            │ + 리니지 메타데이터      │
                            └──────────────────────┘
```

**기존 flux-n8n ETL과의 차이:**

| 항목 | flux-n8n (기존) | IMSP (신규) |
|------|----------------|-------------|
| 패러다임 | ETL (변환 후 적재) | ELT (적재 후 변환) |
| 원본 보존 | 변환된 데이터만 저장 | Raw 원본 MinIO에 보존 |
| 메타데이터 | 수동 매핑 | AI 자동 추출 (EXIF, GPS, NER) |
| 콘텐츠 분석 | 없음 | OCR + 엔티티 추출 + 임베딩 |
| 스케일 | 단일 프로세스 | K8s + Argo Workflow (DAG) |
| 재사용 | `kg/etl/pipeline.py` | 코어 로직 재사용, 오케스트레이션 교체 |

### 3.5 노드 분류 체계

IMSP 워크플로우의 노드는 6종으로 분류된다.

| # | 노드 유형 | 역할 | 구체 예시 |
|---|----------|------|----------|
| 1 | **트리거 노드** | 워크플로우 실행 시점 제어 | 수동 실행, 스케줄(Cron), 웹훅, 앱 이벤트, 에러 이벤트 |
| 2 | **액션 노드** | 외부 시스템 작업 수행 | HTTP/GraphQL 호출, 메시지 전송, 파일 I/O, DB CRUD, Execute Command, Docker 실행 |
| 3 | **커넥터 노드** | 외부 시스템 연동 | SaaS/API 연계, SFTP/클라우드 스토리지, SQL DB, OAuth2/OIDC/API Key 인증 위임 |
| 4 | **제어 노드** | 흐름 제어 | If/Switch 분기, Merge 병합, Split in Batches 반복, Wait 지연, Error 처리 |
| 5 | **변환 노드** | 데이터 형식 변환 | 멀티모달(텍스트/수치/테이블/이미지/비디오/음성) → 해사 데이터(궤적/레이더/CCTV/VHF/전자해도/항해술/법령) 변환/매핑 |
| 6 | **특수 노드** | AI/고급 기능 | Skills, MCP, LLM 호출, AI 모델(선박 탐지/OCR/NER), 유틸리티 |

**해사 도메인 특화 변환 노드 예시:**

| 입력 (멀티모달) | 변환 | 출력 (해사 데이터) |
|----------------|------|------------------|
| AIS NMEA 메시지 | NMEA 디코딩 + 정규화 | 선박 궤적 (TrackSegment) |
| 레이더 영상 | 선박 탐지 (YOLO) | 탐지 결과 + 바운딩박스 |
| CCTV 스트림 | 프레임 추출 + 객체 인식 | 선박 ID + 상태 |
| 전자해도 (S-100) | S-100 파싱 + 온톨로지 매핑 | 해역/항로/수심 엔티티 |
| 기상 데이터 (GRIB2) | GRIB2 파싱 + 격자 보간 | WeatherCondition 엔티티 |
| 법령 문서 (PDF) | OCR + NER + 임베딩 | Regulation 엔티티 + 벡터 |
| VHF 음성 | STT + 화자 분리 | 통신 기록 + 엔티티 추출 |

## 4. 연차별 상세 로드맵

### 4.1 1차년도 (2026): 설계 + 개념 검증

> **공식 목표**: 대화형 해사서비스 설계 플랫폼의 운영 개념 정의 및 설계

#### 4.1.1 목표

1차년도는 전체 플랫폼의 **아키텍처를 확정**하고, 핵심 구성요소의 **PoC(개념검증)** 를 수행하며, 해상교통 도메인의 **온톨로지를 재설계**하는 해이다. flux-n8n의 기존 KG 엔진을 flux-platform 모노레포로 이식하고, Keycloak/K8s 등 인프라 기반을 구축한다.

#### 4.1.2 6대 축별 세부 개발 내용

**축 1: 플랫폼 운영 기술**
- K8s 클러스터 초기 구성 (dev/staging 환경)
- Keycloak Realm 설정 + OIDC 연동 PoC
- Neo4j Community 5.x 클러스터 배포 (단일 인스턴스)
- Prometheus + Zipkin 기본 모니터링 구축
- flux-n8n `kg/` → flux-platform `core/` 이식 (import 경로 변경 + 1,148 테스트 통과 확인)

**축 2: 통합개발환경**
- VueFlow 기반 워크플로우 캔버스 PoC (노드 CRUD + 엣지 연결)
- 노드 분류 체계 6종 정의서 작성
- 전역 대화창 UI 프로토타입 (LLM 연동 없이 UI만)

**축 3: 자원 저장소**
- MinIO 배포 + 버킷 정책 설정
- Suredata Lab ↔ flux-platform REST API 명세 v1 확정

4차년도(2단계 1차)는 MVP를 **프로덕션 수준으로 고도화**하고, 해사서비스 개발의 **표준 프로세스**를 정립한다. 성능 최적화, 확장성 강화, UX 개선에 집중하며, 민간 지원을 7건으로 확대한다.

## 5. 온톨로지 재설계 전략

### 5.1 기존 온톨로지 분석 (127 엔티티 → 왜 재설계?)

flux-n8n의 기존 해사 온톨로지(`maritime/ontology/maritime_ontology.py`)는 **127개 엔티티 레이블, 83개 관계 타입, 29개 속성 정의 그룹**으로 구성되어 있다. 이 온톨로지는 1차년도 PoC를 위해 광범위한 해사 도메인을 빠르게 커버하기 위해 설계되었으나, IMSP 플랫폼에서는 다음과 같은 이유로 **완전 재설계**가 필요하다.

| 문제 | 상세 | 영향 |
|------|------|------|
| **범위 불일치** | 기존 온톨로지는 해사 전체를 광범위하게 커버하나, KRISO 요구는 **해상교통 도메인**부터 시작 | 불필요한 엔티티가 KG 복잡도 증가, 쿼리 성능 저하 |
| **데이터 소스 부재** | 기존 엔티티 중 실제 데이터 소스가 확보된 것은 30% 미만 | 빈 스키마가 사용자 혼란 유발 |
| **시간조건부 미지원** | 기존 온톨로지에 validFrom/validTo 등 temporal 속성 부재 | "3월에만 통항 금지" 같은 시간 추론 불가 |
| **ELT 패러다임 미반영** | Raw 데이터 → 메타데이터 → 엔티티 추출 흐름 미고려 | 데이터 리니지 추적 불완전 |
| **S-100 미매핑** | IHO S-100 표준과의 매핑 부재 | 전자해도 데이터 연동 어려움 |
| **Suredata 연계 미고려** | 데이터 마트/서빙 계층과의 인터페이스 부재 | 양방향 데이터 흐름 설계 불가 |

**재사용 가능한 부분:**

그럼에도 불구하고 기존 온톨로지의 다음 부분은 재설계 시 참조/재사용한다:

- 엔티티 분류 체계 (PhysicalEntity, SpatialEntity, TemporalEntity, InformationEntity, ObservationEntity) → 상위 카테고리 유지
- 핵심 관계 패턴 (DOCKED_AT, ON_VOYAGE, OBSERVED_BY 등) → 관계 명명 규칙 계승
- 속성 타입 정의 (PropertyDefinition 클래스) → `kg/ontology/core.py` 그대로 사용
- OWL 변환 파이프라인 (`maritime/n10s/owl_exporter.py` + `kg/n10s/`) → 파이프라인 로직 재사용

### 5.2 KRISO 요구 기반 재설계 방향

KRISO 미팅(2026.03.18)에서 확인된 요구사항을 기반으로 온톨로지를 재설계한다.

**재설계 원칙:**

1. **데이터 주도 (Data-Driven)**: 실제 확보 가능한 데이터 소스가 있는 엔티티만 정의
2. **해상교통 우선 (Maritime Traffic First)**: AIS 기반 해상교통부터 시작, 점진 확장
3. **시간조건부 (Temporal)**: 모든 규칙/제약에 validFrom/validTo 속성 필수
4. **ELT 친화적**: Raw → Metadata → Entity 3계층 리니지 추적 가능한 구조
5. **S-100 호환**: IHO S-100 표준 엔티티와 1:1 매핑 가능한 구조
6. **다층 지식 (Multi-Layer Knowledge)**: 정적 지식 / 동적 관측 / 추론 결과 계층 분리

**재설계 온톨로지 상위 구조 (초안):**

```

├── StaticKnowledge (정적 지식 계층)
```

### 5.5 OWL ↔ Neo4j 동기화 전략

기존 flux-n8n에서 식별된 이슈 중 하나인 **"OWL ↔ Python 온톨로지 동기화 자동화 없음"** 을 IMSP에서 해결한다.

**동기화 파이프라인:**

```
┌──────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ Protege      │     │ OWL Exporter     │     │ Neo4j            │
│ (OWL 2 설계)  │────→│ (Python)          │────→│ (Property Graph)  │
│              │     │ maritime.ttl 생성  │     │ n10s 임포트       │
│ 온톨로지      │     │                  │     │                  │
│ 편집기        │     │ (기존 코드 재사용:  │     │ 스키마 검증 +      │
│              │     │  owl_exporter.py)  │     │ 인덱스 자동 생성   │
└──────┬───────┘     └────────┬─────────┘     └────────┬─────────┘
       │                      │                        │
       └──────────────────────┼────────────────────────┘
                              │
                    ┌─────────▼──────────┐
                    │ CI/CD 자동 동기화    │
                    │ ─────────────────  │
                    │ 1. OWL 변경 감지    │
                    │ 2. Python 모델 생성  │
                    │ 3. Neo4j 스키마 갱신 │
                    │ 4. 테스트 자동 실행   │
                    │ 5. 불일치 시 알림    │
                    └────────────────────┘
```

**동기화 구성요소:**

| 구성요소 | 설명 | 구현 시점 |
|---------|------|----------|
| OWL → Python 코드 생성기 | .ttl 파일에서 Python dataclass 자동 생성 | 2차년도 |
| Python → Neo4j 스키마 동기화 | Ontology 객체에서 Cypher 제약조건/인덱스 자동 생성 | 1차년도 (기존 `init_schema.py` 확장) |
| OWL → Neo4j 직접 동기화 (n10s) | `kg/n10s/importer.py`로 OWL 직접 임포트 | 1차년도 (기존 코드 재사용) |
| CI/CD 파이프라인 | 온톨로지 변경 시 자동 검증 + 배포 | 2차년도 |
| 스키마 버전 관리 | 온톨로지 버전 태깅 + 마이그레이션 스크립트 | 2차년도 |

### 7.2 API 인터페이스 설계 (REST)

**Suredata → IMSP (데이터 주입):**

```
POST /api/v1/ingest/raw
  Body: { source, dataType, rawPayload, metadata }
  → MinIO에 Raw 저장 + 메타데이터 자동 추출 시작

POST /api/v1/ingest/metadata
  Body: { assetId, metadata: { ... } }
  → KG에 메타데이터 노드 생성/갱신

POST /api/v1/ingest/entities
  Body: { assetId, entities: [ { type, properties, relationships } ] }
  → KG에 엔티티 직접 적재

POST /api/v1/ingest/model-registry
  Body: { modelId, name, version, metrics, servingEndpoint }
  → AI 모델 자산을 KG에 등록

POST /api/v1/ingest/mart-sync
  Body: { martId, schema, sampleData }
  → 데이터 마트 스키마/메타데이터를 KG에 동기화
```

**IMSP → Suredata (결과 조회):**

```
GET  /api/v1/query/nl?q={자연어질의}
  → Text2Cypher → Cypher → KG 조회 → 자연어 응답

GET  /api/v1/query/cypher
  Body: { query, params }
  → Cypher 직접 실행

GET  /api/v1/lineage/{assetId}
  → 자산의 전체 리니지 체인 조회

GET  /api/v1/ontology/schema
  → 현재 온톨로지 스키마 (엔티티/관계/속성 목록)

GET  /api/v1/ontology/search?q={검색어}
  → 온톨로지 엔티티/관계 의미 검색

POST /api/v1/model/invoke
  Body: { modelId, input }
  → Suredata 모델 서빙 호출 (프록시)
```

### 7.4 공동 개발 영역

Suredata Lab과 공동으로 개발해야 하는 영역과 협업 방식을 정의한다.

| 공동 영역 | 인사이트마이닝 책임 | Suredata 책임 | 협업 방식 |
|----------|-------------------|-------------|----------|
| API 명세 | API 스키마 정의 (OpenAPI 3.0) | API 구현 검증 | 공동 리뷰 후 확정 |
| 데이터 품질 규칙 | KG 품질 게이트 규칙 정의 | DW 품질 규칙 정의 | 규칙 공유 + 교차 검증 |
| 모델 메타데이터 표준 | KG 모델 스키마 정의 | 모델 레지스트리 표준 | 공동 스키마 정의 |
| 보안 정책 | Keycloak Realm 설계 | DW 접근 제어 | Keycloak 통합 |
| 모니터링 | K8s + KG 메트릭 | DW + 모델 서빙 메트릭 | Prometheus 통합 대시보드 |
