# 제1장. 과업 이해

## 1.1 과업 목적

본 과업은 선박해양플랜트연구소(KRISO)가 추진하는 『디지털서비스 기술 실용화 지원』 과제의 일환으로, **대화형 해사서비스 플랫폼을 위한 지식그래프 모델 설계 연구**를 목표로 합니다.

해사 분야의 다양한 멀티모달 데이터(해양관측, 선박정보, 전자해도, 기상, 항해 등)에 대한 지식그래프 모델을 설계하고, 데이터 간 의미적 관계를 체계화하여, 향후 자연어 기반 데이터 탐색 및 AI 서비스의 기반을 구축하는 것을 목적으로 합니다.

## 1.2 5개년 로드맵에서의 위치

본 과업은 총 5개년 2단계 과제의 **1단계 1차년도**에 해당하며, 플랫폼의 운영 개념 정의 및 지식그래프 모델 설계를 수행합니다.

| 단계 | 연차 | 목표 | 본 과업 |
|------|------|------|--------|
| **1단계** | **1차년도** | **운영 개념 정의 및 설계 (KG 모델 설계)** | **← 본 용역** |
| 1단계 | 2차년도 | 데이터·모델 의미 체계화 + 구성 요소 개발 (그래프 DB 구축 및 탐색 프로토타입) | |
| 1단계 | 3차년도 | 통합 연계 → 플랫폼 MVP | |
| 2단계 | 4차년도 | 플랫폼 고도화 + 표준 개발 프로세스 | |
| 2단계 | 5차년도 | 안정화 + 민간 연구개발 지원 | |

따라서 본 과업의 설계 결과물은 2차년도 그래프 DB 구축 및 자연어 기반 탐색 기능 개발에 **직접 활용 가능한 수준**으로 산출되어야 합니다.

# 제2장. 수행 방법론

## 2.1 연구 요구사항

### 2.1.1 [REQ-001] 해사 데이터 현황 조사 및 의미적 연관성 분석

> **산출물:** 해사 데이터 현황 분석 보고서 (계약 후 2개월 이내)

#### 수행 접근법

본 요구사항은 지식그래프로 구축할 대상 데이터의 **실체를 파악**하고, 데이터 간 **의미적 연결 가능성**을 분석하는 것입니다. 단순한 목록 조사가 아닌, KG 변환 관점에서의 정밀 분석을 수행합니다.

#### 세부 수행 내용

**1) 연구소 내부 및 외부 데이터 제공처별 현황 조사**

| 조사 대상 | 조사 항목 | 수행 방법 |
|---------|---------|---------|
| KRISO 시험시설 8종 (해양공학수조, 선형시험수조, 빙해수조, 심해공학수조, 파력시험장, 고압챔버, 캐비테이션터널, 선박운항시뮬레이터) | 데이터 유형, 포맷, 갱신 주기, 저장 현황, 관리 체계 | 시설 담당자 인터뷰 + 데이터 샘플 분석 |
| 외부 기관 (해수부, 조사원, 측위정보원, 기상청, 해경, 자율운항선박실증센터) | 제공 데이터 목록, API/포맷, 접근 방식, 라이선스 | 공개 포털 조사 + 기관 협의 |
| 멀티모달 데이터 (궤적, 레이다, CCTV, VHF, 전자해도, 항해술, 법령) | 유형별 분류, 메타데이터 구조, 데이터 용량/갱신 주기 | 데이터 프로파일링 도구 활용 |

**2) 데이터 간 의미적 연관 관계(Semantic Link) 식별**

데이터셋 간 연결 가능성을 공통 식별자 기반으로 분석합니다.

```
[AIS 데이터] ──MMSI──→ [선박 등록 정보]
     │                      │
     IMO 번호               IMO 번호
     │                      │
     ▼                      ▼
[해양사고 데이터] ←─항만코드─→ [항만 정보]
     │                      │
     위치(lat/lon)           위치(lat/lon)
     │                      │
     ▼                      ▼
[기상 관측 데이터] ←─해역코드─→ [전자해도]
```

- **공통 식별자 분석:** MMSI, IMO 번호, Call Sign, 항만 코드(UN/LOCODE), 해역 코드 등 데이터셋 간 조인 키(Join Key) 도출
- **외래키(FK) 관계 매핑:** 데이터 소스 간 논리적 의존성 분석 (예: 사고 데이터 → 선박 데이터 → 항만 데이터)
- **엔티티 연결 규칙:** 동일 엔티티의 이종 데이터 소스 간 매칭 규칙 정의

**3) 시공간 데이터의 정합성 및 해상도 분석**

| 분석 항목 | 내용 | 산출물 |
|---------|------|---------|
| 좌표계 정합성 | 동적 데이터(AIS, 센서)와 정적 데이터(해도, 법령) 간 좌표계 통일 방안 | 좌표계 변환 규칙 |
| 시간 기록 형식 | 데이터별 타임스탬프 포맷, 타임존 분석 | 시간 정규화 규칙 |
| 갱신 주기 동기화 | 실시간(AIS) vs 배치(법령) 데이터의 동기화 전략 | 갱신 주기 매트릭스 |
| 시공간 인접성 기준 | KG 노드 및 엣지 생성 시 시공간적 인접성 판단 기준 | 인접성 정의서 |

**4) 지식그래프 변환을 위한 데이터 프로파일링**

- 데이터별 메타데이터 구조, 포맷, 갱신 주기 및 품질 현황 진단
- 결측치 처리 및 논리적 오류(Data Integrity)가 그래프 구조에 미치는 영향 분석
- 비정형 데이터(법령, 지침 등) 내 핵심 엔티티 및 계층 구조 추출(NER/Relation Extraction) 가능성 진단

**5) 지식그래프 적용 시 예상 이슈 및 대응 전략**

| 이슈 | 영향 | 대응 전략 |
|-----|------|---------|
| 대용량 시계열 데이터(AIS) | 노드/엣지 폭발, 쿼리 성능 저하 | 시계열 DB 분리 + KG는 요약 노드만 |
| 보안 등급별 접근 제어 | 노드/속성 수준 접근 제한 필요 | RBAC 기반 노드별 접근 제어 설계 |
| 이기종 데이터 포맷 | 매핑 복잡도 증가 | 표준 중간 포맷(JSON-LD) 정의 |

#### 당사 보유 역량 활용

- **크롤러 4종** (KRISO 논문, 해양사고, 해양기상, 시설) 활용하여 외부 공개 데이터 프로파일링을 즉시 수행
- **엔티티 해석기** (entity_resolution/) 활용하여 이종 데이터 소스 간 엔티티 매칭 규칙 검증

### 2.1.2 [REQ-002] KG 구축 대상 자산(Asset) 선정 및 로드맵 수립

> **산출물:** 자산 선정 보고서 (해사 데이터 현황 분석 보고서에 포함)

#### 수행 접근법

대화형 해사서비스 플랫폼에서 지식그래프로 관리할 멀티모달 데이터, 모델, 소프트웨어 자산의 유형과 범위를 정의하고 전략적 우선순위를 도출합니다.

#### 세부 수행 내용

**1) 플랫폼 자산 유형 정의 및 상세 범위 선정**

| 자산 유형 | 하위 분류 | 예시 |
|---------|---------|-----|
| **데이터 자산** | 선박 정적 정보 | 선명, IMO, 선종, 총톤수, 선급 |
| | 동적 궤적(AIS) | 위치, 속도, 침로, 선수방위 |
| | 환경 정보(기상/해도) | 풍속, 파고, 시정, 수심, 해류 |
| | 비정형 문서(법령/매뉴얼) | 해사안전법, COLREG, 항만규정 |
| **모델 자산** | AI 분석 모델 | 궤적 이상탐지, 충돌 위험 예측 |
| | 시뮬레이션 모델 | 내항성능, 저항추진, 쇄빙 |
| | 알고리즘 | 경로 최적화, 선석 배정 |
| **인프라/워크플로우** | 컴퓨팅 노드 | GPU 서버, K8s 클러스터 |
| | 데이터 처리 파이프라인 | ETL, 스트리밍, 배치 처리 |
| | 서비스 에이전트 워크플로우 | 자연어 QA, 보고서 자동 생성 |

**2) 지식그래프 편입 기준 및 우선순위 도출**

3가지 평가 축으로 자산의 KG 편입 우선순위를 결정합니다.

| 평가 축 | 평가 방법 | 가중치 |
|--------|---------|--------|
| **연결 밀도(Centrality)** | 타 데이터와의 연관 관계가 높아 지식화 시 파급효과가 큰 핵심 자산 식별 | 40% |
| **활용성 및 시급성** | 대화형 서비스 시나리오(사고 대응, 경로 최적화)에서 빈번하게 참조되는 자산 우선순위화 | 35% |
| **데이터 품질 적합성** | 지식화가 가능한 수준의 데이터 정제도 및 신뢰성을 확보한 자산 선별 | 25% |

**3) 자산 간 관계 유형(Relationship) 사전 정의**

| 관계 유형 | 설명 | 예시 |
|---------|------|------|
| **계층 관계 (Taxonomy)** | 도메인 분류 체계에 따른 상하위 관계 | 해상교통 > 항만물류 > 항만안전 > 선박안전 |
| **참조/의존 관계** | 모델 학습에 사용된 데이터, 서비스 실행에 필요한 인프라 간 의존성 | AI 모델 ─TRAINED_ON→ 데이터셋 |
| **시공간 관계** | 특정 위치나 시간대에 동시 발생하는 사건과 객체 간의 연관성 | 선박 ─NEAR→ 항만, 사고 ─OVERLAPS→ 기상경보 |

**4) 단계별 구축 로드맵 수립**

| 단계 | 기간 | 목표 | 자산 범위 |
|------|------|------|---------|
| **1단계 (PoC)** | 1차년도 | 개념 검증을 위해 선정된 핵심 자산 기반 최소 지식 모델(Target Subset) 정의 | 핵심 데이터 3종 + 시험시설 메타데이터 |
| **2~3차년도 확장** | 2~3차년도 | 전사적 자산 통합 및 실시간 데이터 연계 확장을 위한 중장기 로드맵 | 전체 데이터/모델/인프라 자산 |

### 2.1.3 [REQ-003] 그래프 DB 기술 조사 및 최적 솔루션 평가

> **산출물:** 그래프 DB 기술 비교 분석 보고서

#### 수행 접근법

해사 멀티모달 데이터의 효율적 저장, 탐색, 추론을 위한 그래프 DB 기술을 조사하고 PoC 수행을 위한 최적의 솔루션을 선정합니다.

#### 세부 수행 내용

**1) 그래프 데이터 모델 및 엔진 유형 비교 분석**

| 비교 항목 | LPG (Labeled Property Graph) | RDF (Resource Description Framework) |
|---------|------|------|
| 데이터 모델 | 노드-엣지에 속성(Property) 부여 | 주어-술어-목적어 트리플(Triple) |
| 쿼리 언어 | Cypher, GQL(ISO 표준) | SPARQL |
| 대표 제품 | Neo4j, Memgraph, TigerGraph | Apache Jena, Stardog, GraphDB |
| 직관적 모델링 | 우수 (속성 기반) | 보통 (트리플 분해 필요) |
| 쿼리 성능 | 우수 (인덱스 기반) | 보통 (조인 기반) |
| 온톨로지 추론 | 제한적 (플러그인) | 우수 (OWL/RDFS 내장) |
| LLM 통합 | 우수 (Text-to-Cypher) | 제한적 (Text-to-SPARQL 난이도 높음) |
| **해사 도메인 적합성** | 복잡한 관계망 표현에 적합 | 표준 및 온톨로지 기반 의미 추론에 적합 |

**2) 해사 특화 기술 지표 기반 솔루션 평가 기준**

| 평가 지표 | 세부 내용 | 측정 방법 |
|---------|---------|---------|
| **시공간 쿼리 지원** | 선박 궤적 및 위치 데이터 처리를 위한 공간 인덱싱 및 공간 연산 기능 | 공간 쿼리 벤치마크 (Point-in-Polygon, Distance) |
| **대규모 데이터 처리량** | 수백만 건 이상의 센서 및 AIS 데이터를 지연 없이 적재(Ingestion)하고 조회 | 적재 TPS, 쿼리 응답 시간 측정 |
| **스키마 유연성** | 새로운 데이터 자산(모델, 워크플로우 등) 추가 시 기존 데이터 구조에 영향을 주지 않는 유연한 확장성 | 엔티티/관계 추가 시 성능 영향 테스트 |

**3) 라이선스 및 운영 환경 적합성 검토**

| 검토 항목 | 세부 내용 |
|---------|---------|
| 오픈소스 vs 상용 | 기술 지원, 비용(TCO), 기능 차이 비교 분석 |
| K8s 연동 | 연구소 기존 인프라(K8s 기반 플랫폼)와의 연동 용이성 및 컨테이너화 지원 여부 확인 |
| 커뮤니티 및 생태계 | 개발자 생태계 규모, 한국어 자료, LangChain/LLM 통합 생태계 |

**4) 최적 솔루션 제안 및 구축 가이드라인 도출**

- PoC 수행을 위한 최적의 그래프 DB 최종 선정 및 선정 사유 명시
- 선정된 DB에 최적화된 데이터 모델링 기법 및 쿼리 튜닝 가이드라인 제시

#### 당사의 접근

당사는 **Neo4j Community Edition 5.x** 기반의 KG 엔진을 이미 구축·운영한 경험을 보유하고 있으며, CypherBuilder(Fluent Cypher 쿼리 빌더), 공간 인덱스(Point Index), 벡터 인덱스(HNSW), 풀텍스트 인덱스 등 Neo4j의 핵심 기능을 실증적으로 검증한 바 있습니다. 이를 기반으로 LPG vs RDF의 객관적 비교와 함께, 해사 도메인에 최적화된 **하이브리드 전략**(LPG 주 저장소 + RDF 표준 교환 계층)을 제안합니다.

### 2.2.2 [DES-002] 지식그래프 물리 스키마 설계

### 2.2.3 [DES-003] 멀티모달 데이터 지식화 및 적재(ETL) 파이프라인 설계

**1) 데이터 소스별 추출(Extract) 및 인터페이스 설계**

- 비정형 텍스트에서의 엔티티 및 관계 추출(Entity/Relation Extraction) 프로세스 설계

**3) 그래프 DB 적재(Load) 및 성능 최적화 전략**

| 적재 유형 | 전략 | 성능 목표 |
|---------|------|---------|
| 초기 대량 적재(Bulk Loading) | UNWIND + MERGE 배치, 트랜잭션 크기 최적화 | 10만 노드/분 |
| 실시간 증분 적재 | CDC(Change Data Capture) 기반 증분 반영 | 1,000 TPS |
| 무결성 검증 | 제약 조건 + 트랜잭션 기반 검증 프로세스 | 적재 시 100% 검증 |

**4) 파이프라인 운영 및 모니터링 설계**

- 적재 작업 상태, 지연 시간, 처리량 모니터링 기능 설계

### 2.2.4 [DES-004] 지식 탐색 및 복합 질의 아키텍처 설계

> **산출물:** 탐색 아키텍처 설계서

#### 세부 수행 내용

**1) 지식 탐색 및 경로 분석 알고리즘 설계**

- 특정 자산(데이터, 모델 등) 간의 연관성 및 영향도를 파악하기 위한 최단 경로(Shortest Path), 가변 길이 경로 탐색 로직 설계
- 엔티티 간의 중심성(Centrality) 및 유사도 분석 알고리즘 적용 방안 설계

**2) 도메인 특화 복합 질의(Complex Query) 시나리오 설계**

| 시나리오 | 질의 예시 | 기술 요소 |
|---------|---------|---------|
| 사고 관련 법령 탐색 | "특정 구역 내 사고 선박의 관련 법령 및 대응 모델 추출" | 시공간 필터링 + 다단계 경로 탐색 |
| 선박 이력 분석 | "부산항 입항 컨테이너선의 최근 6개월 사고 이력" | 시간 범위 + 엔티티 타입 필터 |
| 모델 의존성 분석 | "특정 AI 모델이 학습한 데이터셋과 관련 실험" | 참조 관계 체인 탐색 |

- 시공간 조건과 속성 필터링이 결합된 복합 쿼리 템플릿 정의

**3) 지식그래프 API 및 인터페이스 아키텍처 설계**

- 외부 시스템이나 대시보드에서 지식그래프 데이터를 호출하기 위한 표준 API(GraphQL, REST 등) 인터페이스 설계
- 2차년도 이후 LLM 연동 및 GraphRAG 기반 확장을 고려한 모듈형 아키텍처 설계

**4) 지식 시각화 및 UI/UX 인터페이스 설계**

- 노드와 엣지의 관계를 직관적으로 파악할 수 있는 그래프 시각화 레이아웃 및 필터링 UI 설계
- 탐색 결과의 텍스트 및 테이블 기반 출력 인터페이스 설계

**5) 쿼리 성능 최적화 설계**

- 복잡한 그래프 탐색 시 응답 성능을 보장하기 위한 쿼리 튜닝 및 캐싱 전략 수립

#### 당사 보유 역량 활용

- `core/kg/query_generator.py` — 다중 언어(한국어/영어) 쿼리 생성기


### 1차년도 직접 효과

- 해사 분야 멀티모달 데이터의 **체계적 현황 파악** 및 지식화 가능성 검증
- KRISO 맞춤형 **해사 도메인 온톨로지** 및 **지식그래프 모델** 확보
- PoC를 통한 **기술적 실현 가능성(Feasibility)** 입증
- 2차년도 그래프 DB 구축에 **즉시 활용 가능한 설계 자산** 확보

### 5개년 과제 기여

- 1차년도 설계 결과를 기반으로 2차년도 그래프 DB 구축 및 탐색 프로토타입 개발 **가속화**
- 확장 가능한 온톨로지 설계로 3~5차년도 플랫폼 고도화 시 **재설계 비용 최소화**
- 국제 표준(S-100, IHO) 정합성 확보로 **해외 시스템 연동 기반** 마련
